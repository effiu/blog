Redis GeoHash And Scan

Posted on 2019-08-24 17:02 In redis Views: Reading time ≈ 5 mins.

Redis在3.2版本后增加了地理位置GEO模块，我们可以用Redis实现“附近的人”、“附近的店”等等功能。

在Redis的日常运维中，有时候需从Redis中找出符合特定条件的key列表手动处理数据。keys指令用来列出所有符合条件的key。

其使用非常简单，但是有两个明显的缺点：

  • 没有offset、limit参数，一次性吐出所有满足条件的key，若结果太多，会非常不方便处理问题。
  • keys是遍历算法，复杂度是O(n)，若实例中有千万级的key，就会导致Redis服务卡顿，所有Redis的其他指令都会被延后甚至卡顿，因为Redis是单线程程序，顺序执行所有指令，其他指令必须等当前keys指令执行完了才可以继续。

Redis HyperLogLog和布隆过滤器

Posted on 2019-08-22 17:49 Edited on 2019-08-22 21:53 In redis Views: Reading time ≈ 8 mins.

若需要统计网站网页的UV和PV数据，如何实现呢？

PV我们需要给每个网页加一个独立的Redis计数器就可以，Key为网页+日期组成。

UV则需要去重，一个用户一天无论请求多少次都只计算一次，这就要求每个网页请求都要带上用户唯一标识，最简单的方法使用set集合存储当天所有访问过此页面的用户Id。sadd统计用户标识，scard可以取出集合大小。但是当访问量大时，比如几千万的UV，那就就需要很大的set集合统计，浪费空间。若页面非常多，所需要的存储空间非常大。

Redis提供了HyperLogLog数据结构就是用来解决这种统计问题的。HyperLogLog提供不精确的去重计数方案，其标准误差是0.81%，这样的精确度已经完全满足UV统计需求了。HyperLogLog是Redis的高级数据结构，非常有用，HyperLogLog数据结构来进行估数，它非常有价值，可以解决很多精确度不高的统计需求。

但是我们无法知道HyperLogLog数据结构内是否存在某个值，因为其为提供相关方法。若有个需要判断值是否在Redis中存在的业务场景，如何处理呢？例如：新闻客户端中，每次推荐都要去掉那些已经看过的内容，新闻客户端如何去重呢？

  • 服务端有统计用户访问记录，直接从用户历史记录中进行筛选，当用户量大，新闻又多的情况下，性能会很差。需要频繁的在数据库中进行exists操作。
  • 缓存，将历史记录缓存起来，这回浪费很大的存储空间，且存储空间会随着时间线性增长
  • 布隆过滤器(Bloom Filter)，专门用来解决去重问题，且还会节省90%以上的空间，但存在误判的情况。
