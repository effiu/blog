并发工具类

Posted on 2019-05-27 23:07 Edited on 2019-09-02 00:54 In concurrent Views: Reading time ≈ 3 mins.

JDK并发包中提供了几个非常有用的并发工具类。CountDownLatch、CyclicBarrier和Semaphore工具类提供了一种并发流程控制的手段。Exchange工具类则提供了在线程间交换数据的一种手段

1. CountDownLatch

允许一个或者多个线程等待其他线程完成操作。

与join()方法类似，join()用于让当前线程等待join线程执行结束。其实现原理是不停检查join线程是否存活，存活则让当前线程永远等待。以下是公平锁与非公平锁测试中CountDownLatch使用方法

CountDownLatch的构造函数接收一个int类型的参数作为计数器。countDown()方法，计数器会减1，await()方法会阻塞当前线程。计数器等于0时await()停止阻塞当前线程。

CountDownLatch不可能重新初始化或者修改CountDownLatch对象的计数器的值。

2. CyclicBarrier
  1. 简介

CyclicBarrier字面意思是可循环使用的屏障。其是让一组线程到达一个屏障(也可以叫做同步点)时被阻塞，直到最后一个线程到达屏障时，屏障才会解除，所有线程才会继续运行。

CycliBarrier默认构造方法是CyclicBarrier(int parties)，其参数表示屏障拦截的线程数量，每个线程调用await()方法通知CyclicBarrier我已经到达屏障，然后阻塞线程，知道所有线程到达屏障。

CyclicBarrier(int parties, Runnable barrierAction)，用于线程到达屏障时，优先执行barrierAction方便处理复杂业务场景。

  1. 应用场景

CyclicBarrier可以用于多线程计算数据，最后合并计算结果的场景。

  1. 与CountDownLatch区别

CountDownLatch的计数器只能使用一次，CyclicBarrier的计数器可以使用reset()方法重置。

CyclicBarrier还有一些其他方法：

  • getNumberWaiting()获得CyclicBarrier阻塞的线程数量
  • isBroken()方法用来了解阻塞线程是否被中断
3. Semaphore

Semaphore(信号量)是用来控制同时访问特定资源的线程数量，它通过协调各个线程以保证合理的使用公共资源。

  1. 应用场景

Semaphore可以用作流量控制，特别是共用资源有限的应用场景，比如数据库连接。Semaphore(int permits)接受一个整型的数字，表示可用许可证数量。

  • acquire(): 获取一个许可证
  • release(): 归还一个许可证
  • tryAcquire(): 尝试获取一个许可证
  • int availablePermits(): 返回此信号量中当前可用许可证数
  • int getQueueLength: 返回正在等待获取许可证的线程数
  • boolean hasQueuedThreads(): 是否有线程正在等待获取许可证
  • protected protected void reducePermits(): 减少reduction个许可证
  • protected Collection getQueuedThreads(): 返回所有等待获取许可证的线程集合
4. Exchanger

Exchanger（交换站）是一个用于线程间协作的工具类。用于进行线程间的数据交换。

Exchanger可以用于遗传算法和校对工作

若交换的两个线程有没有执行exchange()方法，则会一直等待

exchange(V x, long timeout, TimeUnit unit)设置最大等待时长

1.ConcurrentHashMap的实现原理与使用

java.util.concurrent.ConcurrentHashMap是线程安全且高效的HashMap。

1. 为什么要使用ConcurrentHashMap

并发编程中HashMap可能会导致程序死循环，HashTable效率低下。

  • JDK 7会导致死循环，JDK 8并不会

HashMap线程不安全，多线程环境下，HashMap进行put操作会导致死循环，导致CPU利用率接近100%。多线程会导致HashMap的Entry链表形成环形数据结构，Entry的next节点永远不为空，就会产生死循环获取Entry。

Hashtable效率低下。其使用synchronized保证线程安全。

ConcurrentHashMap的锁分段技术可以有效提升并发访问率。Hashtable效率低下是因为其只有一把锁，导致所有线程竞争一把锁。ConcurrentHashMap所使用的锁分段技术，将数据分成一段一段的存储，然后每一段使用一把锁，当线程访问一段数据的时候，其他段的数据并不会被锁住，也可以被其他线程访问。

2. ConcurrentHashMap的结构

通过ConcurrentHashMap类图分析ConcurrentHashMap结构。

ConcurrentHashMap是由Segment数组结构和HashEntry数组结构组成。Segment是一种可重入锁(ReentrantLock)，其和HashMap结构类似，是一种数组和链表结构，每个Segment守护一个HashEntry数组里的额元素，对HashEntry数组的元素修改时，必须首先获得与它对应的Segment锁。

3. ConcurrentHashMap的初始化

ConcurrentHashMap的初始化是通过initialCapacity、loadFactor、concurrentLevel等几个 参数完成初始化Segment数组、段偏移量segmentShift、段掩码segmentMask和segment里的HashEntry数组实现的。

阻塞队列是一个支持两个附加操作的队列，阻塞插入和阻塞移除。

2. Java里的阻塞队列

  • ArrayBlockingQueue

    • 由数组结构组成的有界阻塞队列
    • 先进先出的原则对元素排序
    • 默认情况不保证线程公平，为了保证公平，会降低吞吐量

  • LinkedBlockingQueue

    • 由链表结构组成的有界阻塞队列
    • 默认最大长度是Integer.MAX_VALUE，按照先进先出的规则进行元素排序

  • PriorityBlockingQueue

    • 支持优先级排序的无界阻塞队列
    • 默认采用自然顺序升序排列
    • 也可以自定义compareTo()指定元素排序规则，也可以构造PriorityBlockingQueue时指定Comparator对元素进行排序。不能保证同优先级元素的顺序

  • DelayQueue

    • 使用优先级队列实现的无界阻塞队列，使用PriorityQueue实现

    • 队列中的元素必须实现Delay接口，创建元素时可以指定延迟时间。

    • DelayQueue非常有用，有以下应用场景

      • 缓存系统，使用DelayQueue保存缓存元素的有效期，一旦成功获取表明缓存到期
      • 定时任务调度，使用DelayQueue保存当天执行的任务和执行时间，一旦获取到任务就开始执行

    • 实现Delay接口

      • 在对象创建的时候，初始化基本数据，使用time记录当前对象延迟时间，sequenceNumber标识元素在队列中的先后顺序。

      • 实现getDelay()接口，返回当前元素还需要延时的时间，单位ns

      • 实现compareTo()方法指定元素的顺序。

    • 实现延时阻塞队列，获取元素方法(take()、poll())部分源码

  • SynchronousQueue

    • 不存储元素的阻塞队列，每个put()都必须等待take()操作，否则不能继续添加元素
    • 支持公平访问，默认情况下采用非公平策略访问队列
    • 其吞吐量高于LinkedBlockingQueue和ArrayBlockingQueue

  • LinkedTransferQueue

    • 由链表结构组成的无界阻塞队列
    • 多个tryTransfer()和transfer()方法
      • transfer()，如果当前消费者正在等待接收元素，transfer可以把生产者传入的元素立刻transfer给消费者，若没有消费者则将元素放在队列的tail节点，并等待该元素被消费了才返回
      • tryTransfer()，是用来试探生产者传入的元素是否能直接传给消费者。支持超时返回。

  • LinkedBlockingDeque：由链表结构组成的双向阻塞队列

4. Frok/Join框架

Fork/Join的基本原理、算法、设计方式、应用与实现

1. Fork/Join框架

是一个并行执行任务的框架，把大任务分割成若干个小任务，最终汇总每个小任务结果后得到大任务结果的框架。

2. 工作窃取算法

工作窃取算法是指某个线程从其他队列里窃取任务来执行

  • 优先：充分利用线程进行并行计算，减少了线程之间的竞争
  • 缺点：当双端队列只存在一个任务时，扔存在竞争，此时会消耗更多系统资源。例如，多个双端队列和多个线程。
  • 缺点：当双端队列只存在一个任务时，扔存在竞争，此时会消耗更多系统资源。例如，多个双端队列和多个线程。
3. Fork/Join 框架的设计
  1. 分割任务，有一个fork类把大任务分割成小任务，若小任务仍然很大，则可以继续分割
  2. 执行任务并合并结果，子任务分别放在双端队列里，然后启动几个线程分别从双端队列里获取任务执行。子任务执行结果统一放到一个队列里，启动一个线程从队列里拿数据，合并结果

ForkJoinTask，使用ForkJoin框架，必须要创建一个ForkJoin任务。它提供在任务中执行fork()和join()操作的机制。Fork/Join框架有两个子类

  • RecursiveAction: 用于没有返回结果的任务
  • RecursiveTask: 用于有返回结果的任务

ForkJoinPool：ForkJoinTask需要通过ForkJoinPool来执行，任务分割出的子任务会添加到当前线程所维护的双端队列中，进入队列的头部，当一个工作线程队列里暂时没有任务使，它会随机从其他工作线程的队列尾部获取一个任务。

双重检查锁定与延迟初始化

Posted on 2019-05-22 23:55 Edited on 2019-09-02 00:54 In concurrent Views: Reading time ≈ 4 mins.

在Java多线程中有时候需要采用延迟初始化来降低初始化类和创建对象的开销。双重检查锁定是常见的延迟初始化技术。但这是一个错误的用法。

1. 双重检查锁定

Java程序中，有时候会需要推迟一些高开销的对象初始化操作，只有在使用这些对象时才需要初始化，此时就可以采用延迟初始化，正确使用线程安全的延迟初始化需要一些技巧，否则很容易出问题。

上述非线程安全实例中，线程A可能会看到未完成初始化的实例instance(2会解释出现的原因)。一种办法是对getInstance()做同步处理public synchronized static String getInstance()，synchronized将导致性能开销，若该方法被多个线程频繁调用，将会导致执行性能的下降。当不会被多个线程频繁调用时，该方案可行。

因为synchroinzed存在巨大的性能开销，所以有了**双重检查锁定(Double-Check Lock)**，以此来降低同步的开销。

第一个if判断检查instance，若不为null，则不需要执行加锁和初始化操作。可以大幅降低synchronized带来的性能开销。有以下好处：

  • 多线程试图在同一时间创建对象时，通过加锁来保证只有一个线程创建对象
  • 对象创建之后，getInstance()也不需要获取锁

当代码执行到第4行，代码读取到instance不为null时，instance引用的对象可能还没有完成初始化

2. 问题的根源

代码instance = new String("initialization");创建了一个对象。可以分解为3步伪代码：

  • memory = allocate(); // 1：分配对象内存空间
  • ctorInstance(memory); // 2：初始化对象
  • instance = memory; // 3：设置instance指向分配的内存空间

伪代码中，2和3之间可能会被重排序，所以执行3后对象可能扔未被初始化。

《The Java Language Specification，Java SE 7 Edition》中规定，所有线程在执行Java程序时必须要遵守intra-thread semantics。intra-thread semantics保证重排序不会改变单线程内的程序执行结果。即允许单线程内不会改变程序执行结果的重排序。

多线程并发的情况见下图：

当多线程执行时，双重检查机制可能会出现如上情况，B线程看到一个未被初始化的对象。

时间 线程A 线程B
t1 A1：分配对象内存空间
t2 A3：设置instance指向内存空间

  • 不允许2和3的重排序
  • 允许2和3重排序，但不允许被其他线程可见

本质上是通过禁止伪代码2和3排序实现的。

4. 基于类初始化的解决方案

JVM在类的初始化阶段（即在Class被加载后，且被线程使用之前），会执行类的初始化。在初始化期间，JVM会去获取一个锁，这个锁可以同步多个线程对同一个类的初始化。基于这个特性，实现另一种线程安全的延迟初始化加载方案(Initialization On Demand Holder idiom)

假设两个线程并发执行getInstance()方法，首次执行getInstance()方法的线程将导致InstanceHolder被初始化，如下是执行示意图：

实质就是，允许创建对象3步中的2和3重排序，但是不允许非构造线程看到这个重排序。初始化类，包括执行这个类的静态初始化和初始化在这个类中声明的静态字段。根据Java语言规范，首次发生以下任意情况时，类或者接口类型T将被立即初始化。

  • T是一个类，且一个T类型的实例被创建
  • T是一个类，且T中声明的一个静态方法被调用
  • T中声明的一个静态字段被赋值
  • T中声明的静态字段被使用，且这个字段不是一个常量字段
  • T是一个顶级类(Top Level Class)，且一个断言语句嵌套在T内部被执行

Java语言是多线程的，多个线程可能在同一时刻尝试初始化同一个类或接口，所以Java中初始化一个类或接口时，需要做细致的同步处理。

Java语言规范规定，对于每一个类或接口C，都有一个唯一的初始化锁LC与之对应。从C到LC的映射，由JVM的具体实现去自由实现。JVM在类初始化期间会获取这个初始化锁，并且每个线程至少获取一次保证这个类已经被初始化完成。

Java初始化一个类或者接口的处理过程如下（省略了与多线程无关的部分，为了更好说明初始化过程中的同步机制，把类的初始化过程分为了5个阶段）：

  1. 通过在Class对象上同步（即获取Class对象的初始化锁），来控制类或接口的初始化。这个获取锁的线程会一直等待，直到当前线程能够获取到这个初始化锁。

基于类的初始化方案实现代码更简单，但是基于volatile的双重检查锁定方案不仅仅可以对静态字段实现延迟初始化，还可以实现对实例字段的延迟初始化。

字段延迟初始化降低了初始化类或创建实例的开销，但增加了访问被延迟初始化的字段的开销。

大多时候，正常初始化优于延迟初始化。但若需要对实例字段使用线程安全的延迟初始化，则使用volatile的延迟初始化方案；若需要对静态字段使用线程安全的延迟初始化，使用基于类初始化的方案。

2. Condition实现分析

ConditionObject是AbstractQueuedSynchronizer内部类，Condition操作需要获取相关联的锁，每个Condition对象都包含着一个等待队列，该队列是等待/通知功能的关键

  1. 等待队列

等待队列是一个FIFO队列，队列中每个节点都包含一个线程引用，该线程就是在Condition上等待的线程，Condition.await()方法会使该线程释放锁、构造成节点加入到等待队列中进入等待状态。实际上，同步队列和等待队列中节点类型都是同步器的静态内部类AbstractQueuedSynchronizer.Node。

一个Condition包含一个等待队列，Condition包含首节点和尾节点，Condition.await()会将线程构造成节点放入到尾节点，进入等待状态。新增节点只需要，将尾节点的nextWaiter指向新增节点，更新Condition的lastWaiter即可。调用await()的线程必定持有锁，同时也保证了线程安全以及等待队列的线程安全

Lock可以有多个Condition，所以可以存在多个等待队列，即拥有一个同步队列和多个等待队列。

  1. 等待

Condition.await()会使线程进入等待队列并释放锁、线程状态变为等待状态，当从await()返回时，线程一定重新获得了锁。当调用await()方法时，线程从同步队列的首节点，进入到等待队列的尾节点。如下代码所示：

3. 重入锁

重入锁(ReentrantLock)，就是支持线程重进入锁。表示该锁能够支持一个线程对资源的重复加锁。还支持获取锁时的公平锁和非公平选择。synchronized关键字隐式的支持重进入。

ReentrantLock在调用lock()方法时，已经获取到锁的线程，允许再次调用lock()方法获取锁而不被阻塞。

公平锁指的是在绝对时间上，先对锁进行获取的请求一定先被满足，即按照同步队列的FIFO规则获取锁。公平锁效率比非公平锁效率低，因为公平锁的上下文切换频率高，造成了更大的系统开销，但是公平锁可以减少“饥饿”发生的概率。

ReentrantLock实现重进入和公平性获取锁的特性如下：

    • 线程再次获取到锁，即识别获取锁的线程是否是当前占有锁的线程
    • 锁的最终释放，重复n次获取锁，那么需要n次释放，ReentrantLock使用计数表示当前锁被重复获取的次数，n=0表示锁已成功释放

4. 读写锁

同一时刻只允许一个线程访问，这种锁属于排他锁。读写锁在同一时刻可以允许多个读线程访问，但在写线程访问时，所有的读线程和其他写线程均被阻塞。读写锁维护了一对锁（读锁和写锁）。读写锁保证了写操作对读操作的可见性，还简化了读写交互场景。

写锁被获取到后，所有后续非当前线程的读写操作均被阻塞，当写锁释放后，其他线程的操作才能继续执行。一般情况下，读写锁比排他锁好，因为大多数读场景多于写，具有更好的并发量和吞吐量。Java提供的读写锁是java.util.concurrent.locks.ReentrantReadWriteLock。其特性见下图表：

2.2 AQS分析

从实现角度分析同步器是如何完成线程同步的，包括同步队列、独占式同步状态获取与释放、共享式同步状态获取与释放、超时获取同步状态等同步器核心数据结构与模板方法。

  1. 同步队列

    同步器依赖内部的同步队列(FIFO的双向队列)完成同步状态的管理，线程获取同步状态失败时，同步器会将当前线程以及等待状态信息构造成一个节点(java.util.concurrent.locks.AbstractQueuedSynchronizer.Node)并将其加入到等待队列，阻塞当前线程，同步状态释放时，会把首节点中的线程唤醒，使其再次尝试获取同步状态。

    java.util.concurrent.locks.AbstractQueuedSynchronizer.Node，用来保存获取同步状态失败的线程引用、等待状态以及前驱和后继节点。节点的属性类型与名称如下：

    属性类型与名称 描述
    int waitStatus 等待状态，包含以下状态：
    ①CANCELLED，值为1，在同步队列中等待的线程等待超时或被中断，需要从同步队列中取消等待，节点进入该状态将不会变化
    ②SINGAL，值为-1，后继节点的线程处于等待状态，当前节点的线程如果释放了同步状态或者被取消，将会通知后继节点，使后继节点的线程得以运行

    ④PROPAGATE，值为-3，表示下一次共享式同步状态获取将会无条件地被传播下去
    ⑤INITIAL，值为0，初始状态

1. 锁

锁是用来控制多个线程访问共享资源的方式，一个锁可以防止多个线程同时访问共享资源，有些锁可以允许多个线程并发的访问共享资源，例如读写锁。

Java SE 5 之后新增了java.util.concurrent.locks.Lock接口用来实现锁功能。它提供了与synchronized关键字类似的同步功能，但是在使用时需要显式获取和释放锁，虽然没有了隐式获取和释放锁的便捷性，但是拥有了获取与释放锁的可操作性、可中断的获取锁以及超时获取锁等多种synchronized没有的同步特性。其使用如下：

java.util.concurrent.locks.Lock接口提供的synchronized不具备的特性。

特性 描述
非阻塞地获取锁 线程尝试获取锁，若此时没有线程获取到，则成功获取并持有锁
能被中断地获取锁 与synchronized不同，获取到锁的线程能够响应中断，中断时会抛出异常并释放锁
超时获取锁 在指定时间之前获取锁，超时后放弃获取锁并返回

2. 队列同步器

队列同步去java.util.concurrent.locks.AbstractQueuedSynchronizer（同步器），是用来构建锁或者其他同步组件的基础框架，其内部使用了一个int成员变量表示同步状态，通过内置的FIFO队列完成资源获取线程的排队工作。

同步器的主要使用方式是继承，子类通过继承同步器并实现其抽象方法来管理同步状态。子类被推荐定义为自定义同步组件的静态内部类，同步器自身没有实现任何同步接口，仅仅定义了若干同步状态的获取和释放的方法供自定义同步组件使用，即可以支持独占式获取同步状态，也可以支持共享式获取同步状态。

同步器实现锁的关键，是在锁的实现中聚合同步器，利用同步器实现锁的语义。

  • 锁是面向使用者的，定义了使用者与锁交互的接口，隐藏了实现细节
  • 同步器面向的是锁的实现者，它简化了锁的实现方式，屏蔽了同步状态管理、线程的排队、等待与唤醒底层操作。

锁和同步器很好地隔离了使用者和实现者所需要关注的领域。

2.1 队列同步器的接口与示例

同步器的设计是基于模板方法模式的，使用者需要继承同步器并重写指定的方法，随后将同步器组合在自定义同步组件的实现中，并调用同步器提供的模板方法。修改同步状态的3个方法

  • getState()：获取当前同步状态
  • setState(int newState)：设置当前同步状态
  • compareAndSetState(int expect, int update)：使用CAS设置当前状态，可以保证同步状态设置的原子性。

3. 线程池技术以及示例

线程的创建和消亡都是会消耗系统资源，当并发量大时，会频繁创建线程。这会导致频繁的线程上下文切换，增加系统负载。线程池可以解决这类问题，其预先创建若干数量线程，不能由用户直接对线程的创建进行控制，重复使用固定或者较为固定数目的线程完成任务，这样可以减少频繁创建和消亡线程的系统资源开销，即使面对过量任务也可以平缓的劣化。
