Java IO

IO基本类；IO底层核心操作技巧；同/异步 + 阻塞/非阻塞的性能区别； BIO、NIO、AIO区别；

  • NIO 操作Socket时的多路复用？

1. IO

1. File

File类既能代表一个特定文件的名称，又能代表一个目录下一组文件的名称。文档地址，部分api说明如下：

  • File.separator，返回当前系统的文件分隔符”/“、”\“等。
  • list()，返回一个由此抽象路径名表示的目录中的文件和目录字符串组成的数组。
  • list(FilenameFilter filter)，返回一个由此抽象路径名表示的目录中满足指定过滤器的文件和目录组成的数组。FilenameFilter 是一个接口，建议使用匿名内部类的方式实现。

java.nio.file.Files包含对文件，目录或其他类型文件进行操作的静态方法。在大多数情况下，这里定义的方法将委托给相关的文件系统提供程序来执行文件操作。

2. 输入和输出

Java中的流，代表任何有能力产出数据的数据源对象或者有能力接收数据的接收端对象。“流”屏蔽了实际的I/O设备中处理数据的细节。

Java类库中的I/O类分为输入和输出两种：

  • 任何继承自java.io.InputStream或者java.io.Reader派生而来的类都含有名为read()的基本方法，用于读取单个字节或者字节数组。
  • 任何继承自java.io.OutputStream或者java.io.Writer派生而来的类都含有名为write()的基本方法，用于写单个字节或者字节数组。

read()和write()方法的存在并不是为了供使用，而是为了别的类使用，用来提供更有效的接口。实际上，很少使用单一的类创建流对象，而是通过叠加多个对象来提供所期望的功能(装饰器设计模式)。

1. InputStream类型

作用是用来表示从不同数据源产生输入的类。数据源主要包括：

  • 字节数组
  • String对象
  • 文件
  • 管道，工作方式与实际管道相似
  • 一个由其他种类的流组成的序列，便于我们将其合并到一个流内
  • 其他数据源，如Internet连接等。

每种数据源都有对应的InputStream类，FilterInputStream也属于一种InputStream，为装饰器类提供基类，装饰器类可以把属性或者有用的接口与输入流连接在一起。InputStream类型如下：

3. 添加属性和有用接口

4. Reader和Writer

Reader和Writer提供兼容Unicode与面向字符的I/O功能，当我们“字节”层次结构中的类和“字符”层次结构中的类结合使用时，要用到适配器类：InputStreamReader和OutputStreamReader。Reader和Writer主要是为了国际化。

5. RandomAccessFile

RandomAccessFile 适用于由大小已知的记录组成的文件，以便我们可以使用 seek() 将记录从一处转移到另一处，然后读取或者修改记录。文件中记录的大小不一定都相同，只要我们能够确定那些记录有多大以及它们在文件中的位置即可。

RandomAccessFile 不是InputStream或者OutputStream继承层次结构中的一部分。除了实现了DataInput 和 DataOutput 接口（DataInputStream 和 DataOutputStream 也实现了这两个接口）之外，它和这两个继承层次结构没有任何关联。它甚至不使用 InputStrean 和 OutputStream 类中已经存在的任何功能。它是一个完全独立的类，从头开始编写其所有方法的类。

6. 管道流

PipedInputStream、PipedOutputStream、PipedReader及PipedWriter只有在我们开始理解多线程之后才会使用，因为管道流用于线程之间的通信。

7. 标准I/O

标准I/O，其意义在于可以很容易把程序串联起来，一个程序所有标准输出可以称为另外一程序的标准输入。

  • 程序所有输入都可以来自于标准输入
  • 程序所有输出也都可以发送到标准输出
  • 程序所有错误信息都可以发送到标准错误
1. 从标准输入中读取

Java提供了System.in、System.out、System.err。其中System.out、System.err已经被包装成了PrintStream。System.in是一个未被包装加工的InputStream，所以使用System.in之前需要对其进行包装。例如：使用InputStreamReader把System.in包装成Reader。

2. System.out转成PringWriter

System.out是PrintStream，其本质是一个OutputStream，PrintWrite有可以接受OutputStream作为参数的构造器。

3. I/O重定向

Java的System提供了一些静态方法调用，允许我们对标准输入、输入和错误I/O流进行重定向:

  • setIn(InputStream)
  • setOut(PrintStream)
  • setErr(PrintStream)

2. NIO

JDK1.4中的java.nio.*中引入了新的Java I/O类，其目的在于提高速度。实际上旧的IO包已经使用nio重新实现过，以便充分速度提高速度，因此实际上不显式使用nio，也能提高速度。

提高速度在文件I/O和网络I/O(在《Thinking in Enterpise Java》中有说明)中都有可能发生。网络I/O的速度提升体现在NIO的单线程轮询事件的机制。以文件I/O进行说明：速度的提高来自于所使用的结构更接近于操作系统执行I/O的方式：通道和缓冲器。实际过程是当发生I/O操作时，仅仅与缓冲器进行交互，并把缓冲器发送到通道。通道要么从缓冲器中获得数据，要么向缓冲器发送数据。

缓冲区(Buffer)就是在内存中预留指定大小的存储空间用来对输入/输出(I/O)的数据作临时存储，这部分预留的内存空间就叫做缓冲区，其有两个作用：

  • 减少实际的物理读写次数
  • 在创建时就会分配内存，这块区域会一直被重用，减少动态分配和回收内存的次数

唯一与通道交互的缓冲器是ByteBuffer，其是可以存储未加工字节的缓冲器，其是非常基础的类：通过告知分配多少存储空间创建一个ByteBuffer对象，还有一个方法选择集，用于以原始字节形式或基本数据类型输出和读取数据，但是没有办法输出或者读取对象。

旧的Java I/O中修改了三个类，用以产出FileChannel：FileInputStream、FileOutputStream、RandomAccessFile。Reader、Writer字符流不能产生通道(ByteBuffer用以以原始形式或者基本数据类型输入和输出数据)。java.nio.channels.Channels提供了方法用于在通道中产生Reader和Writer。

上述三种流中的getChannel()方法产生了一个FileChannel对象。通道是一种相当基础的东西，可以向它传递用于读写的ByteBuffer，并且可以锁定文件的某些区域用于独占式的访问(fc.lock())。

ByteBuffer的常用方法如下：

  • put和get用于填充或者获取字节或者基本数据类型

  • wrap()用于将已经存在的字节数组包装到ByteBuffer中，其不会不复制byte数据，而是把其作为所产生的ByteBuffer的存储器。

  • allocate用于显式分配ByteBuffer，便于快速移动大量数据

  • read()告知FileChannel向ByteBuffer存储字节，调用flip()让其做好被读取的准备，clear()方法为read()做准备，即当重复使用read是，必须先执行clear()。

  • transferTo()和transferFrom()则允许我们将一个通道和另一个通道直接相连

1. 转换数据

当我们使用nio处理字符时，使用ByteBuffer不太方便，所以可以使用CharBuffer。

缓冲器容纳的是普通的字节，若我们需要把其转换为字符，要么在输入时进行编码，要么在从缓冲器输出时进行解码。

2. ByteBuffer转换为其他类型

ByteBuffer只能保存最基本的字节类型，但是其可以从其所容纳的字节中产生各种不同基本类型值的方法。

  • rewind()用于从返回到数据开始部分。

向ByteBuffer插入基本类型数据的方法有asCharBuffer()、asShortBuffer等获取该缓冲器上的视图，然后使用视图的put()方法。其适用于所有除了asShortBuffer()之外的所有基本类型。使用ShortBuffer的put()方法时需要类型转换。

3. 视图缓冲器

视图缓冲器(view buffer)可以让我们通过某个特定的基本数据类型的视窗查看器底层的ByteBuffer。ByteBuffer依然是实际存储数据的地方。对视图的任何修改都会映射成为对ByteBuffer中数据的修改。

put()和get()是IntBuffer重载后的方法，其最直接访问底层ByteBuffer中的某个整数位置

一旦底层的ByteBuffer通过视图缓冲器填满了整数或其他基本类型时，就可以直接被写到通道中了。视图缓冲器可以把任何数据都转化为某一特定的基本类型。

5. 缓冲器的细节

Buffer由数据和可以高效地访问操纵这些数据的四个索引组成：

  • mark，标记
  • position，位置
  • limit，界限
  • capacity，容量

以下是设置和复位索引以及查询它们值的方法：

  • capacity()：返回缓冲区容量
  • clear()：清空缓冲区，将position设置为0，limit设置为capacity。
  • flip()：将limit设置为position，position设置为0，用于准备从缓冲区中读取已经写入的数据
  • limit()：返回limit()值
  • limit(int lim)：设置limit的值
  • mark()：将mark设置为position
  • position()：返回position的值
  • position(int pos)：设置position的值
  • remaining()：返回（limit-position）
  • hasRemaining()：若存在介于position和limit之间的元素，则返回true

6. 内存映射文件

内存映射文件允许我们创建和修改那些因为太大而不能放入内存的文件。使用内存映射文件，我们可以就可以假定整个文件都放在内存中，可以把其当做非常大的数组访问，即将文件的某一段内容写入到内存中。

MappedByteBuffer是ByteBuffer子类。

尽管BIO在用NIO实现后性能有所提高，但是”映射文件访问”往往可以更加显著地加快速度。

7. 文件加锁

JDK4引入了文件加锁机制，其允许我们同步访问某个作为共享资源的文件。竞争同一文件的两个线程可能在不同的Java虚拟机上；或者一个是Java线程，另一个是操作系统中其他某个本地线程。文件锁对其他操作系统进程是可见的，因为Java的文件加锁直接映射到了本地操作系统的加锁工具。

FileChannel的tryLock()、lock()可以获得整个文件的FileLock。tryLock()是非阻塞式的，其尝试获得锁，若获取失败(其他线程持有相同的锁，其不共享时)，将直接从调用方法返回。lock()是阻塞式的，其要阻塞进程直到获得锁或者调用lock()的线程中断或者调用lock()的通道关闭。release()可以释放锁。

tryLock(long position, long size, boolean shared)或者lock(long posotion, long size, boolean share)用于对文件的一部分加锁。

对文件加共享锁或者独占锁必须由底层操作系统提供。如操作系统不支持共享锁，那么就只能使用独占锁。

3. BIO、NIO、AIO比较

  • IO概念的比较
  • BIO是传统的java.io包，是基于流模型实现的，交互方式是同步、阻塞，即在输入或者输出流时，在完成之前，线程会一直阻塞，其调用顺序非常可靠。特点：代码简单、直观；IO效率和扩展性低，容易成为应用性能瓶颈。
  • NIO是Java4引入的java.nio包，提供了Channel、Selector、Buffer等新的抽象，可以构建多路复用、同步非阻塞IO程序，同时提供了更接近操作系统底层高性能的数据操作方式。
  • AIO(Asynchronous IO)是Java7引入的包，是NIO的升级版本，提供了异步非阻塞的IO操作方式，异步IO是基于事件和回调机制实现的，也就是应用操作之后会直接返回，不会堵塞，后台处理完成后，操作系统会通知相应的线程进行后续操作。

NIO需要使用者线程不停的轮询IO对象，来确定是否有数据准备好可以读了; AIO则是在数据准备好之后，才会通知数据使用者，不需要不停地轮询。

组合方式 性能分析
同步阻塞 最常用、最简单。IO性能一般，CPU大部分时间处于空闲状态。java.io
同步非阻塞 提示IO性能的常用手段，尤其网络IO是长连接，同时传输数据不是很多的情况下，提升非常有效，但是会增加CPU消耗。需要考虑增加的IO性能能否补偿CPU消耗。
异步阻塞 在分布式数据库中经常使用，应用于主从复制上。异步阻塞可以提升网络IO的效率
异步非阻塞 在一些非常负责的分布式情况下使用，集群之间的消息同步机制使用异步非阻塞机制。适合要传多分相同的数据到集群中不同的机器，同时传输数据流不大但却很频繁的情况。
