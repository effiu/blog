Dubbo常见知识点。

1. 框架的分层方式

  • 共有10层。左侧淡蓝背景为服务消费方使用的接口；右侧淡绿色背景为服务提供方使用的接口；中轴线上是双方都使用的接口。
  • 从上至下分为10层，各层均为单向依赖，右边的黑色箭头代表层之间的依赖关系。Service和Config均为API，其余为SPI(Service Provider Interface,一种服务发现机制)。

各层说明：

  • Service业务层：业务代码的接口与实现。

  • Config配置层：对外配置接口，以ServiceConfig、ReferenceConfig为中心，可以直接初始化配置类，也可以通过Spring解析配置生成配置类。

  • Proxy服务代理层：服务接口透明代理，生成服务的客户端Stub和服务端Skeleton，以ServiceProxy为中心，扩展接口为ProxyFactory。Dubbo会为provider和consumer生成代理，代理之间进行网络通信。

  • registry注册中心层：封装服务地址的注册与发现，以服务URL为中心，扩展接口为RegistryFactory、Registry、RegistryService。

  • cluster路由层：封装多个提供者的路由及负载均衡，并桥接注册中心，以Invoker为中心，扩展接口为Cluster、Directory、Router、LoadBalance。

  • monitor监控层：RPC调用次数和时间监控，以Statistics为中心，扩展接口为MonitorFactory、MonitorService、Monitor。

  • protocol远程调用层：封装RPC调用，以Invocation、Result为中心，扩展接口为Protocol、Invoker、Exporter

  • exchange信息交换层：封装请求响应模式，同步转异步，以Request、Response为中心，扩展接口为Exchanges、ExchangeChannel、ExchangeClient、ExchangeServer。

  • transport网络传输层：抽象mima和netty为统一接口，以Message为中心，扩展接口为Channel、Transport、Client、Server、Codec

  • serialize数据序列化层，可复用的一些工具，扩展接口为Serialization、ObjectInput、ObjectOutput、ThreadPool

2. 中的Filter

  • Dubbo的异常处理核心为Provider的异常过滤器ExceptionFilter对调用结果各种情况的处理。具体见dubbo异常处理
  • 参数校验功能，通过参数校验过滤器 alidationFilter 来实现。在 Dubbo Provider和Consumer都可生效。dubbo参数校验
  • Dubbo通过CacheFilter过滤器，提供结果缓存的功能，且既可以适用于Consumer也可以适用于Provider。dubbo Cache

3. 优雅停机

Dubbo是通过JDK的ShutdownHook来完成优雅停机的，所以如果用户使用kill -9 PID等强制关闭指令，是不会执行优雅停机的。kill PID，才会执行。

服务提供方优雅停机过程：

  • 停止时，先标记为不接收新请求，新请求过来时直接报错，让客户端重试其它机器。
  • 然后，检测线程池中的线程是否正在运行，如果有，等待所有线程执行完成，除非超时，则强制关闭。

服务消费方的优雅停机过程：

  • 停止时，不再发起新的调用请求，所有新的调用在客户端即报错。

  • 然后，检测有没有请求的响应还没有返回，等待响应返回，除非超时，则强制关闭。

设置优雅停机：dubbo.service.shutdown.wait=15000。

若ShudownHook不能生效，可以自行调用，使用tomcat等容器部署的场景，建议通过扩展ContextListener等自行调用ProtocolConfig.destroyAll();实现优雅停机。

5. 序列化方式

对应serialize数据序列化层，Dubbo 目前支付如下 7 种序列化方式：

  • Hessian2 ：基于 Hessian 实现的序列化拓展。dubbo协议的默认序列化方案。
  • Dubbo ：Dubbo 自己实现的序列化拓展。 《精尽 Dubbo 源码分析 —— 序列化（二）之 Dubbo 实现》 。
  • Kryo ：基于kryo实现的序列化拓展。 《Dubbo 用户指南 —— Kryo 序列化》
  • FST ：基于FST实现的序列化拓展。 《Dubbo 用户指南 —— FST 序列化》
  • JSON ：基于 Fastjson 实现的序列化拓展。
  • NativeJava ：基于 Java 原生的序列化拓展。
  • CompactedJava ：在 NativeJava 的基础上，实现了对 ClassDescriptor 的处理。

6. 负载均衡策略

对应Cluster路由层的loadBalance组件。Dubbo有4钟负载均衡策略。默认使用random随机调用策略。

  • Random LoadBalance，随机，按照权重设置随机概率。
  • RoundRobin LoadBalance，轮询，按照公约后的权重设置轮询比率。存在慢的提供者请求累积问题。
  • LeastActive LoanBalance，最少活跃调用数，相同活跃的随机，活跃数指调用计数差。慢的提供者收到更少的请求。
  • ConsistentHash LoadBalance，一致性hash，相同参数的请求总是发到同一提供者。

7. 集群容错策略

对应Cluster路由层的Cluster组件，Dubbo有6种容错策略：

  • failOver Cluster，失败自动切换，出现失败时，重试其他服务器。通常用于读操作，重试会带来延迟。可以设置重试次数retries=2
  • failfast Cluster，快速失败，只发起一次调用，失败立即报错。通常用于非幂等性的写操作。
  • failsafe Cluster，失败安全，出现异常时，直接忽略。通常用于写入审计日志等操作。
  • failback Cluster，失败自动恢复，后台记录失败请求，定时重发。通常用于消息通知操作。
  • forking Cluster，并行调用多个服务器，只要一个成功即返回。用于实时性较高的读操作，会浪费更多的资源。设置最大并行次数fork=2
  • broastcast Cluster，广播调用所有提供者，逐个调用，任意一台报错则报错。用于通知所有提供者更新缓存或者日志等本地资源。

8. 动态代理策略

对应Proxy代理层。Dubbo动态代理使用了Javassist和JDK两种方式。默认是Javassist，可以通过SPI方式切换为JDK。

Javassist代理的字节码bytecode生成代理类性能最好。

9. SPI的设计思想

JAVA拾遗–关于SPI机制

  • JDK 标准的 SPI 会一次性实例化扩展点所有实现，如果有扩展实现初始化很耗时，但如果没用上也加载，会很浪费资源。Dubbo 有很多的拓展点，且每个扩展点有多个实现
  • 如果扩展点加载失败，连扩展点的名称都拿不到了。
  • 增加了对扩展点 IoC 和 AOP 的支持，一个扩展点可以直接 setter 注入其它扩展点。

10. 服务监控和惯例

  • Dubbo 管理平台 + 监控平台，Dubbo Monitor监控平台。Dubbo admin基于注册中心的管理平台。

  1. dubbo服务降级

    比如说服务A调用服务B，结果服务B挂掉了。服务A再重试几次调用服务B，还是不行，那么直接降级，走一个备用的逻辑，给用户返回响应。Dubbo中实现降级有两种方式：

    • Dubbo自带的服务降级功能，并不能实现现代微服务的熔断器的功能
    • 引入支持服务降级的组件
      • Alibaba Sentinel
      • Netflix Hystrix，已停止维护，且与Dubbo集成度不高。

  2. dubbo限流

    Alibaba Sentinel。
