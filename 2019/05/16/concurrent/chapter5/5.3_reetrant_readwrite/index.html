重入锁与读写锁

Posted on 2019-05-16 00:26 Edited on 2019-09-02 00:54 In concurrent Views: Valine: Reading time ≈ 7 mins.
3. 重入锁

重入锁(ReentrantLock)，就是支持线程重进入锁。表示该锁能够支持一个线程对资源的重复加锁。还支持获取锁时的公平锁和非公平选择。synchronized关键字隐式的支持重进入。

ReentrantLock在调用lock()方法时，已经获取到锁的线程，允许再次调用lock()方法获取锁而不被阻塞。

公平锁指的是在绝对时间上，先对锁进行获取的请求一定先被满足，即按照同步队列的FIFO规则获取锁。公平锁效率比非公平锁效率低，因为公平锁的上下文切换频率高，造成了更大的系统开销，但是公平锁可以减少“饥饿”发生的概率。

ReentrantLock实现重进入和公平性获取锁的特性如下：

  • 实现重进入要满足以下两点
    • 线程再次获取到锁，即识别获取锁的线程是否是当前占有锁的线程
    • 锁的最终释放，重复n次获取锁，那么需要n次释放，ReentrantLock使用计数表示当前锁被重复获取的次数，n=0表示锁已成功释放

4. 读写锁

同一时刻只允许一个线程访问，这种锁属于排他锁。读写锁在同一时刻可以允许多个读线程访问，但在写线程访问时，所有的读线程和其他写线程均被阻塞。读写锁维护了一对锁（读锁和写锁）。读写锁保证了写操作对读操作的可见性，还简化了读写交互场景。

写锁被获取到后，所有后续非当前线程的读写操作均被阻塞，当写锁释放后，其他线程的操作才能继续执行。一般情况下，读写锁比排他锁好，因为大多数读场景多于写，具有更好的并发量和吞吐量。Java提供的读写锁是java.util.concurrent.locks.ReentrantReadWriteLock。其特性见下图表：

特性 说明
公平性选择 支持非公平(默认)和公平的锁获取方式，吞吐量非公平由于公平
重进入 支持重入，读线程获取锁之后，能够再次获取读锁。写线程获取写锁之后可以再次获取读锁写锁
锁降级 遵循获取写锁、获取读锁再释放写锁的次序，写锁降级为读锁（其他线程不会立即获取写锁）
1. 读写锁接口

java.util.concurrent.locks.ReadWriteLock定义了读锁和写锁的两个方法，即readLock()和writeLock()方法。java.util.concurrent.locks.ReentrantReadWriteLock还提供了一些用于监控内部工作状态的方法。见下表：

方法名称 描述
int getReadLockCount() 返回当前锁被获取的次数。因为可以重复获取锁，所以不等于读锁的线程数
int getReadHoldCount() 返回当前线程获取读锁的次数。使用ThreadLocal保存次数
boolean isWriteLocked() 判断写锁是否被获取
int getWriteHoldCount() 返回当前线程写锁被获取的次数(只能有一个线程获取写锁)

java.util.concurrent.locks.ReentrantReadWriteLock实现：读写状态的设计、写锁的获取与释放、读锁的获取与释放以及锁降级。

  1. 读写状态设计

读写锁的自定义同步器需要在同步状态上维护多个读线程和一个写线程状态，所以需要“按位切割使用“这个状态变量，高16位表示读，低16位表示写。

使用位运算获取当前读和写各种的状态。假设同步状态为S，那么S & 0000FFFF，按位与运算将高位抹去即写状态。读状态等于S>>>16（无符号补0右移）。写状态增加即S+1，读状态增加即S+(1<<16)。

锁降级，值的是写锁降级为读锁。即把持住当前拥有的写锁，再获取到读锁，随后是否写锁的过程。

上例中，volatile update，当数据变更后，update变量被设置为false，其他线程访问此方法会感知到变化，但是只要一个线程获取到写锁，其他线程会被阻塞在读锁或写锁的lock()方法上。当前线程获取写锁完成后再次获取读锁，随后释放写锁，完成锁降级。

锁降级中，写锁释放之前获取读锁是为了保证其他写线程不会感知不到写操作。

ReentrantReadWriteLock不支持锁升级（获取读锁、获取写锁、是否读锁的过程）。因为锁升级过程不能保证可见性，读锁已被多个线程获取，获取写锁后，其他线程感知不到写锁的操作。
