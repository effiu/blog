Redis HyperLogLog和布隆过滤器

Posted on 2019-08-22 17:49 Edited on 2019-08-22 21:53 In redis Views: Valine: Reading time ≈ 8 mins.

若需要统计网站网页的UV和PV数据，如何实现呢？

PV我们需要给每个网页加一个独立的Redis计数器就可以，Key为网页+日期组成。

UV则需要去重，一个用户一天无论请求多少次都只计算一次，这就要求每个网页请求都要带上用户唯一标识，最简单的方法使用set集合存储当天所有访问过此页面的用户Id。sadd统计用户标识，scard可以取出集合大小。但是当访问量大时，比如几千万的UV，那就就需要很大的set集合统计，浪费空间。若页面非常多，所需要的存储空间非常大。

Redis提供了HyperLogLog数据结构就是用来解决这种统计问题的。HyperLogLog提供不精确的去重计数方案，其标准误差是0.81%，这样的精确度已经完全满足UV统计需求了。HyperLogLog是Redis的高级数据结构，非常有用，HyperLogLog数据结构来进行估数，它非常有价值，可以解决很多精确度不高的统计需求。

但是我们无法知道HyperLogLog数据结构内是否存在某个值，因为其为提供相关方法。若有个需要判断值是否在Redis中存在的业务场景，如何处理呢？例如：新闻客户端中，每次推荐都要去掉那些已经看过的内容，新闻客户端如何去重呢？

  • 服务端有统计用户访问记录，直接从用户历史记录中进行筛选，当用户量大，新闻又多的情况下，性能会很差。需要频繁的在数据库中进行exists操作。
  • 缓存，将历史记录缓存起来，这回浪费很大的存储空间，且存储空间会随着时间线性增长
  • 布隆过滤器(Bloom Filter)，专门用来解决去重问题，且还会节省90%以上的空间，但存在误判的情况。

1. HyperLogLog

1. 使用方法

HyperLogLog提供了两个指令pfadd和pfcount

  • pfadd：增加计数，与sadd类似
  • pfcount：获取计数，与scard类似

HyperLogLog除了pfadd和pfcount之外，还有pfmerge指令，用于将多个pf计数值累加在一起形成一个新的pf值

HyperLogLog数据结构需要占据一定12K的存储空间，其不适合统计单个用户相关的数据。当计数较小时，其存储空间采用稀释矩阵存储，占用空间小。当计数变大后，稀疏矩阵占用空间超过阈值时就会变为稠密矩阵(12K空间)

矩阵中，若数值为0的元素数目远远多于非0的数目，且非0元素分布没有规律时，则成为稀疏矩阵，反之则为稠密矩阵。非零元素的总数比上矩阵所有元素总数为矩阵的稠密度。

2. 布隆过滤器

布隆过滤器可以理解位一个不精确的set结构，当使用布隆过滤器的contains方法判断对象是否存在时，可能会误判，但也可以把误差控制在很小范围内。布隆过滤器判断某个值存在时，其可能不存在，但当判断为不存在时，那肯定不存在。

其应用到新闻去重中，布隆过滤器能准确判断已经看过的内容。没看过的内容也可能会漏掉(误判)一部分，但是能保证绝大多数新内容都能准确识别。这样就能保证推荐给用户的内容是无重复的。

1. 使用方法

布隆过滤器有两个基本指令，

  • bf.add添加元素，与set集合中sadd类似，但一次只能添加一个元素。
  • bf.exists，查询元素是否存在，用法与set集合的sismember差不多。
  • bf.madd，添加元素，可以一次添加多个。
  • bf.mexists，查询多个元素是否存在，

直接使用布隆过滤器，就会在我们第一次bf.add时使用默认参数构建(如上文的INITIAL_SIZE 100000 ERROR_RATE 0.001)。Redis还提供了自定义参数的布隆过滤器。在bf.add之前使用bf.reserve指令显式创建。若对应的key已存在，bf.reserve报错。其参数：key、error_rate、initial_size。错误率越低，需要的空间越大。initial_size表示预计放入的元素数量，当实际数量超出该值时，错误率会明显上升。

布隆过滤器的initial_size估计的过大，会浪费存储空间，估计的过小，会影响准确率，所以在使用前一定要尽可能精确估计好元素数量，还需要加一定的冗余空间以避免实际元素可能会意外高出估计值很多。error_rate越小，需要的存储空间越大。

3. 实现原理

布隆过滤器对应到Redis的数据结构里就是一个大型的位数组和几个不一样的无偏hash函数，无偏指的是可以把元素的hash值算的比较均匀。

向布隆过滤器中添加key时，会使用多个hash函数对key进行hash算得一个整数索引值然后对位数组长度进行取模运算得到一个位置，每个hash函数都会算得一个不同的位置。再把位数组的这几个位置都置为1就完成了add操作。

向布隆过滤器询问key是否存在时，与bf.add一致，也会把hash的几个位置算出来，看看位数组中这些位置是否为1，只要1个为0，那就说明布隆过滤器中这个key不存在；若都为1，那就说明这个key极有可能存在，并不是肯定存在，因为有可能这些位被置为1是因为其他key存在导致的。位数组越稀疏，准确率越高，越拥挤则准确率越低。

使用布隆过滤器时，不要让实际元素大于初始化大小，当实际元素数开始超出初始化大小时，应对布隆过滤器进行重建，重新分配一个size更大的过滤器，再将所有历史元素批量add进去。error_rate不会因为数量超出就急剧增加，这给我们重建过滤器提供了较为宽松的时间。

4. 占用空间估计

布隆过滤器占用空间有一个简单的计算公式。布隆过滤器有两个参数：

  • n，预计元素的数量
  • f，错误率

由以上两个输入可以得到两个输出：

  • l，需要的存储空间大小，bit
  • k，hash函数的最佳数量，hash函数会直接影响错误率，最佳的数量会有最低的错误率

从公式可以得出：

  • 位数组越长，错误率越低
  • 位数组越长，hash函数越多，越影响计算效率
  • 当一个元素平均需要8bit指纹空间时，错误率大约为2%
  • 错误率为10%，一个元素平均需要4.793 bit
  • 错误率为1%，一个元素需要9.585 bit
  • 错误率为0.1%，一个元素需要14.377 bit

可以发现一个元素仅仅需要15bit就能控制错误率在0.1%。布隆过滤器存储的是元素的指纹，需要15bit，而元素的内容甚至可以达到几百字节，而且元素本身需要一个指针被set集合来引用，所以布隆过滤器的空间优势是非常明显的

目前已经由网站支持布隆过滤器相关计算，例如[布隆计算器](https://krisives.github.io/bloom-calculator/]

5. 其他应用场景

  1. 爬虫系统，对URL进行去重，已经爬过的网页不需要继续爬
  2. NoSQL，当用户来查询某个Row时，可以先通过布隆过滤器过滤掉大量不存在的Row请求，显著降低IO请求
  3. 邮箱系统，垃圾邮件过滤，误判很低
