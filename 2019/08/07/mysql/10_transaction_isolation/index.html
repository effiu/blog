MySQL事务及隔离性

Posted on 2019-08-07 18:56 Edited on 2020-06-07 19:26 In mysql Views: Valine: Reading time ≈ 9 mins.

程序员的大部分工作是把现实世界的业务场景映射到数据库，我们要保证数据库操作符合现实世界中状态转换的规则。但是实际中，现实世界的一个变化，可能对应数据库的多条操作。例如，银行转账（A给B转账10元）：

  • 查询A账户金额
  • A账户余额减去10元
  • A账户修改后的余额写到磁盘
  • 查询B账户余额
  • B账户余额增加10元
  • B账户余额写入到磁盘

如上，现实中的一个不可分割的操作，对应了多个数据库操作，那么我们就需要保证这多个数据库操作不会执行一部分出现问题。

  • 数据库本身的错误
  • 操作系统错误
  • 直接断电
  • ······

1. 事务的概念

1. 数据库的特性

  1. 原子性(Atomicity)

转账是一个不可分割的操作，要么没转，要么转账成功，没有中间的状态。数据库中这种要么全做，要么全不做的规则称为原子性。

  1. 隔离性(Isolation)

现实中，银行的两次转账是互不影响的，例如当“A给B转账10元，但银行单笔限额5元时“，A向B转账过程中，会发生查询余额、存钱、转账等等数据库操作。现实与真实数据库中操作可能分别如下：

所以我们要保证数据库内其他状态转换不会影响到本次状态转换，这个规则称为隔离性。

  1. 一致性(Consistency)

现实世界中存在的约束当然也要在数据库世界中有所体现，若数据库中的数据全部符合现实世界中的约束，则这些数据就是一致的，即一致性。保证数据的一致性需要现实数据库与业务代码一起保证：

  • 数据库本身可以为我们保证一部分一致性需求(数据库本身可以保证一部分现实世界的约束永远有效)。MySQL数据库可以为表建立主键、唯一索引、外键、声明某个列为NOT NULL拒绝NULL值插入等，还可以通过自定义触发器的方式定义一些约束条件保证数据库的一致性。

  • 通过写业务代码，保证一致性。

    现实生活中复杂的需求有很多，由于性能问题不能把所有的需求都交给数据库。例如，复杂的转账操作。数据库中某些操作的原子性和隔离性都是保证一致性的一种手段，在操作执行完成后保证符合所有既定的约束则是一种结果。所以原子性与隔离性都会对一致性产生影响。

  1. 持久性(Durability)

数据库中一个操作完成后，其结果将永久保留，称为持久性。

2. 事务的概念

原子性(Atomicity)、隔离性(Isolation)、一致性(Consistency)和持久性(Durability)，即ACID。需要保证原子性、隔离性、一致性、持久性的一个或者多个数据库操作称之为一个事务(transaction)。事务是一个抽象的概念，具有以下几个状态：

  • 活动的(active)，事务对应的数据库操作正在执行过程中，即该事务处于活动状态。
  • 部分提交(partially committed)，事务中最后一个操作执行完成，但由于操作都在内存中执行，所造成的影响并没有刷新到磁盘，此时事务处于部分提交的状态。
  • 失败的(failed)，事务处于活动中或者部分提交状态时，可能会遇到某些错误（数据库本身的错误，操作系统错误，直接断电等等）而无法继续执行，所以人为停止当前事务的执行，该事务处于失败的状态
  • 中止的(aborted)，当事务处于失败的状态，需要撤销事务对当前数据库造成的影响，即回滚。当回滚操作执行完毕后，数据库会恢复到执行事务之前的状态，该事务处在了中止的状态。
  • 提交的(committed)，处在部分提交的事务将修改过的数据同步到磁盘后，事务处于提交的状态。

事务处于提交的和中止的状态时，事务的生命周期已经结束了，即事务commit或者rollback后，事务结束。

3. MySQL中事务的语法

  • BEGIN [WORK]，代表开启一个事务

  • STARG TRANSACTION，开启一个事务，写法：START TRANSACTION READ ONLY

    • READ ONLY：标识当前事务是一个只读事务
    • READ WRITE：标识当前事务是一个读写事务
    • WITH CONSISTENT SNAPSHOT：启动一致性读

  • COMMIT [WORK]，提交一个事务

  • ROLLBACK [WORK]，手动中止一个事务

4. 支持事务的存储引擎

MySQL中并不是所有的存储引擎都支持事务的功能，目前只有InnoDB和NDB存储引擎支持。

5. 自动提交

autocommit，是一个系统变量，默认为ON，即一个SQL算一个独立的事务，称为自动提交。

当显式使用START TRANSACTION或者BEGIN语句开启一个事务，那么本次事务再提交或者回滚前会关闭自动提交的功能。

6. 隐式提交

当使用START TRANSACTION或者BEGIN语句开启一个事务，或者把系统变量autocommit设置为OFF后，事务不会自动提交，但是如果我们输入某些语句，事务会自动提交，即隐式提交：

  • 定义或修改数据库对象的数据定义语言(DDL)，使用CREATE、ALTER、DROP等语句修改数据库、表、视图、存储过程等等，会隐式提交事务
  • 隐式使用或修改mysql数据库中的表，ALTER USER、CREATE USER、DROP USER、GRANT、RENAME USER、REVOKE、SET PASSWORD等语句
  • 事务控制或关于锁定的语句
    • 事务提交或者回滚前，使用START TRANSACTION或者BEGIN语句开启另一个事务时
    • LOCK TABLES、UNLOCK TABLES等关于锁定的语句
    • LOAD DATA语句批量向数据库中导入数据时
  • MySQL复制的一些语句，START SLAVE、STOP SLAVE、RESET SLAVE、CHANGE MASTER TO等语句时
  • 其他，ANALYZE TABLE、CACHE INDEX、CHECK TABLE、FLUSH、LOAD INDEX INTO CACHE、OPTIMIZE TABLE、REPAIR TABLE、RESET等语句也会隐式提交事务

2. 事务隔离级别

MySQL是一个客户端/服务端架构的软件，对于服务器来说，可以有若干客户端与之连接，每个连接都是一个会话(Session)。每个客户端都可以在会话中发出请求，一个请求语句可以是某个事务的一部分，服务器也可能同时处理多个事务，事务之间需要符合隔离性，理论上某个事务对某个数据进行访问时，其他事务要求排队，当该事务提交后，其他事务才可以继续访问这个数据。但是这样大大影响性能，所以要求既保持事务的隔离性，又让服务器在处理访问同一数据的多个事务时性能尽量高些，这就要求我们舍弃部分隔离性而保证高性能。

1. 准备

2. 事务并发执行的问题

  • 脏写(Dirty Write)，会话B修改了会话A未提交的修改过的数据，事务B回滚。

  • 脏读(Drity Read)，会话A读取到了会话B未提交的事务修改的数据，会话B回滚事务。

  • 不可重复读（Non-Repeatable Read）,事务A只能读到另一个已经提交的事务修改过的数据，且其他事务每对该数据进行一次修改并提交后，该事务都可以查询到最新值，则发生了不可重复读。

  • 幻读（Phantom），事务A先根据某些条件查询出一些记录，之后事务B又向表中插入了符合这些条件的记录，事务A继续以该条件查询时，能把事务B插入的记录也读出来，那么发生了幻读。即在一个事务的两次查询中数据不一致。

不可重复读的重点是修改：同样的条件，你读取过的数据，再次读取出来发现值不一样了
幻读的重点：在于新增或者删除： 同样的条件，第 1 次和第 2 次读出来的记录数不一样

3. SQL中的隔离级别

MySQL中事务并发执行引发的问题中，按照问题严重性分为：脏写 > 脏读 > 不可重复读 > 幻读，上文说的舍弃部分隔离性换取部分性能：设立一些隔离级别，隔离级别越低，越严重的问题就越可能发生，目前SQL标准中有4种隔离级别：

  • READ UNCOMMITTED：未提交读
  • READ COMMITTED：已提交读
  • REPEATABLE READ：可重复读
  • SERIALIZABLE：可串行化

  • SESSION，在会话范围影响
    • 对当前会话的所有后续事务有效

    • 该语句不能再已经开启的事务中间执行

可以在启动MySQL服务器时设置隔离级别，启动参数：--transaction-isolation=SERIALIABLE

3. MVCC原理

1. 版本链

MySQL中InnoDB存储引擎中，其聚簇索引中包含三个隐藏列

  • DB_ROW_ID：非必须，当数据库表中存在主键或者非NULL的UNIQUE键时，不会创建该列
  • DB_TRX_ID：每次一个事务对某条聚簇索引记录进行改动时，都会把该事务的事务id赋值给trx_id隐藏列
  • DB_ROLL_PTR：每次对某条聚簇索引记录进行改动时，都会把旧的版本写入到undo日志中，然后该列就相当于一个指针，可以通过它来找到该记录修改前的信息。

2. ReadView

  • Read UNCOMMITTED隔离级别的事务，由于可以读取到未提交事务修改过的记录，所以直接读取版本链的最新记录。
  • SERIALIZABLE隔离级别的事务，InnoDB使用加锁的方式访问记录
  • READ COMMITTED和REPEATABLE READ隔离级别的事务，都必须保证读到已经提交了的事务修改过的记录，其核心问题是：需要判断一下版本链中哪个版本是当前事务可见的。

针对上述要求有了ReadView概念：

  • m_ids：表示在生成ReadView时当前系统中活跃的读写事务的事务id列表
  • min_trx_id：表示在生成ReadView时当前系统中活跃的读写事务中最小的事务id，也就是m_ids中的最小值
  • max_trx_id：表示生成ReadView时系统中应该分配给下一个事务的id，其并不是m_ids中的最大值
  • creator_trx_id：表示生成ReadView的事务id

当我们访问某条记录时，只需要按照下面步骤判断版本是否可见：

  • 被访问版本的trx_id属性值与ReadView中的creator_trx_id相同，即当前事务访问它自己修改过的记录
  • 若被访问版本的trx_id属性值小于ReadView中的min_trx_id，表示生成该版本的事务在当前事务生成的ReadView前已经提交，即可以被当前事务访问
  • trx_id大于或等于ReadView中的max_trx_id值，表示生成该版本的事务在当前事务生成ReadView后开启，不可以被当前事务访问
  • trx_id在min_trx_id和max_trx_id之间，那么就判断一下trx_id属性值是否在m_ids中，若在则说明创建ReadView时该版本的事务是活跃的，则该版本不可以被访问；若不在，说明创建ReadView时生成该版本的事务已经被提交，则该版本可以被访问

当某个版本对当前事务不可见时，就按照版本链找到上一个版本的数据，继续判断其可见性，直到确认版本可见或者版本链最后一个版本。

5. MVCC小结

综上所述，MVCC(Muti-Version Concurrency Version，多版本并发控制)指的就是READ COMMITTED、REPEATED READ这两种隔离级别的事务在执行普通的SELECT操作时访问记录版本链的过程，这样可以使不同事务的读写、写读操作并发执行，从而提升系统性能。READ COMMITTED和REPEATED READ这两个隔离级别区别就是：生成ReadView的时机不同

  • READ COMMITTED每次进行普通查询操作前都会生成一个ReadView
  • REPEATED READ只在第一次查询操作前生成一个ReadView

insert undo在事务提交后就可以被释放掉了，而update undo由于还需要支持MVCC，不能立即删除。为了支持MVCC，delete mark操作仅仅是在记录上打了一个删除标记，并没有真正删除。当系统确定ReadView的事务不会再访问某些update undo日志以及被打了delete mask的记录后，后台purge线程会把其真正删除掉。
