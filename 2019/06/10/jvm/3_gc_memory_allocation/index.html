垃圾收集器与内存分配策略

Posted on 2019-06-10 00:51 Edited on 2020-04-11 17:57 In jvm Views: Valine: Reading time ≈ 14 mins.

Java与C++之间主要区别：内存动态分配和垃圾收集技术

1. 概述

当需要排查各种内存溢出、内存泄露问题时，当垃圾收集称为系统达到更高并发量的瓶颈时，就需要我们对内存分配与内存回收实施必要的监控与调节。

Java内存运行区域中程序计数器、虚拟机栈、本地方法栈会随着线程而存在和销毁；栈中的栈帧随着方法的进入和退出而执行出栈和入栈操作，每个栈帧中分配多少内存是在类结构中确定的（JIT编译器会进行一些优化），所以这些区域不需要过多考虑回收，它们会随着线程结束或者方法结束被回收。

Java堆和方法区是所有线程共享的，一个接口的多个实现类需要的内存不确定，一个方法的多个分支中需要的内存也不确定，只有在程序运行期才知道创建哪些对象，其内存分配和回收是动态的，所以需要垃圾收集器回收这部分区域。

2. 判断对象是否需要回收

  1. 引用计数器法

    给对象添加一个引用计数器，每当有地方引用对象时，计数器加1；引用失效时，计数器减1，任何时候计数器为0的对象都是可回收的。优点：实现简单，判断效率高。缺点：无法解决对象之间相互循环引用的问题，所以JVM中没有使用引用计数器法。

  2. 可达性分析算法

    通过一系列GC Roots对象作为起始点，从这些节点开始向下搜索，搜索走过的路径为引用链，当一个对象到GC Roots没有任何引用链相连时(GC Roots到这些对象不可达)，则说明此对象不可用。

    Java中可以作为GC Roots的对象包括以下几种：

    • 虚拟机栈中引用的对象
    • 方法区中静态常量属性引用的对象
    • 方法区中常量引用的对象
    • 本地方法栈中JNI(Java Native Interface)引用的对象

  3. 对象引用

    引用计数器法与可达性分析算法，都是根据引用判断对象是否存活。存在这么一类对象：当内存空间足够时，则可以保留在内存中，当内存空间在进行垃圾回收后，内存仍不够使用，则可以抛弃这些对象。Java引用分为：强引用(Strong Reference)、软引用(Soft Reference)、弱引用(Weak Reference)、虚引用(Phantom Reference)，其引用强度依次减弱。

    • 强引用，在代码中普遍存在，Object o = new Object();这类引用都是强引用，只要强引用存在，垃圾收集器永远不会回收引用的对象
    • 软引用，是指一些有用但非必需对象，这些对象在系统将要发生OOM时，会把这些对象列进回收范围之中进行二次回收，若回收后仍没有足够的内存，则抛出OOM
    • 弱引用，描述非必需对象，被弱引用关联的对象只能生存到下一次GC发生之前。发生GC时，必然会回收弱引用关联的对象
    • 虚引用，唯一目的是：在这个对象被收集器回收时收到一个系统通知

  4. 对象的生存与死亡

    即使可达性分析算法中不可达的对象，也可以继续存活下去，这种对象处于第一次标记阶段，要被回收至少要经历两次标记过程。当经过可达性分析后，发现没有与GC Roots相连的引用链，对象将被第一次标记并且进行一次筛选，判断是否有必须执行finalize()方法，若对象没有覆盖finalize()方法，或者finalize()方法已经被虚拟机调用过，虚拟机将认为对象没有必要执行finalize()方法。

    当对象被判定为有必要执行finalize()方法，这个对象将被放入到一个叫做F-Queue队列中，并在稍后由一个虚拟机自动建立的、低优先级的Finalizer线程去执行对象的finalize()方法，但是这个线程并不会等待其执行结束，这是为了防止finalize()方法执行缓慢，或者发生死循环，可能导致队列中其他对象永久处于等待状态，甚至导致整个GC回收缓慢。若对象在finalize()方法中与GC Roots关联(例如，把this关键字赋值给某个类变量或者对象的成员变量)，那么第二次标记时将会被移出”即将回收”的集合，若对象没有与GC Roots关联将会被回收

    实际开发中尽量不要使用finalize()方法，若要回收资源建议使用try-with-resource。

  5. 回收方法区

    Java虚拟机规范中可以不要求虚拟机实现方法区的垃圾收集，且方法区中的垃圾收集一般性价比低。方法区的垃圾收集主要分为两部分：废弃常量和无用的类。

    • 废弃常量回收，与Java堆中的对象回收类似，当没有任何引用关联常量时，在垃圾回收时回收 该常量。常量池中其他类(接口)、方法、字段的符号引用与此类似（static修饰符修饰）。

    • 无用的类，类需要满足以下3个条件才会被标记为“无用的类”，满足以下3个条件后，可以被回收，但是不一定要回收

      • 该类所有实例都被回收，Java堆中不存在该类的实例
      • 该类的ClassLoader已被回收
      • 该类的java.lang.Class对象没有在任何地方被引用，无法在任何地方通过反射访问该类的方法

    大量使用反射、动态代理、CGLib等ByteCode框架、动态生成JSP以及OSGi这类频繁自定义ClassLoader的场景都需要虚拟机具备类卸载的功能，保证方法区不会溢出

3. 垃圾收集算法

  1. 标记—清除算法，分为标记、清除两个阶段。

    • 标记，标记处所以需要回收的对象
    • 清除，标记完成后统一回收被标记的对象

    不足之处：效率低，标记和清除效率很低；空间利用率低，标记清除之后会产生大量不连续的空间碎片，可能会导致以后需要分配大对象时，无法找到足够的连续内存不得不提前出发GC

  2. 复制算法

    解决了效率问题，将可用内存空间分为大小相等的两部分，每次只使用其中一块，当被使用的内存满后，将活着的对象复制到另外一部分去，再把之前的空间一次性全部清空。

    • 优点：每次回收都是对整个内存区域的回收，回收效率高，内存分配时也不存在内存碎片，只需要移动堆顶指针，按顺序分配对象。
    • 缺点：将内存分割为两部分，实际内存利用率低

    商业虚拟机采用这种算法回收新生代。其Eden、From Survivor、To Survivor比例为8:1:1。

    内存分配过程:

    • 对象优先分配在 Eden 区，如果 Eden 区无法分配，那么尝试把活着的对象放到From Survivor中去（Minor GC）
    • 如果From Survivor可以放入，那么放入之后清除Eden区。
    • 如果From Survivor不可以放入，那么尝试把 Eden 和From Survivor的存活对象放到 To Survivor中。
      • 如果 To Survivor可以放入，那么放入 To Survivor之后清除 Eden 和From Survivor，之后再把To Survivor中的对象复制到From Survivor中，保持To Survivor一直为空。
      • 如果 To Survivor 不可以放入，那么直接把它们放入到老年代中，并清除 Eden 和 From Survivor，这个过程也称为分配担保。

  3. 标记—整理算法

    复制算法在对象存活率高的内存区域中会进行较多的复制操作，效率降低，且会浪费内存空间。标记—整理算法：

    • 标记，标记过程与标记—清除的标记过程一样。
    • 整理，让所有存活对象向一端移动，然后直接清除掉边界以外的内存。

  4. 分代收集算法

    根据对象存活周期将内存区域划分为几块：新生代(Eden、From Survivor、To Survivor)和老年代。根据各自特点采用不同垃圾收集算法。

    • 新生代，每次垃圾收集都会有大批对象被回收，所以采用复制算法，只需要付出少量的内存空间
    • 老年代，对象存活、没有额外的空间进行分配担保，标记整理或者标记清除算法。
4. HotSpot算法实现

实际JVM实现垃圾收集相关算法时，需要对算法有严格的考虑，才能保证虚拟机高效运行。

  1. 枚举根节点

    可达性分析中从GC Roots节点寻找引用链，可作为GC Roots节点主要是全局性的引用(常量或类静态属性)与执行上下文(栈帧中的本地变量)，实际应用中很多方法区达到数百兆，若要逐个检查这里的引用，会消耗很多资源。

    可达性分析，还会造成GC停顿，因为分析工作必须在一个能确保一致性的快照中进行(分析期间整个执行系统停顿，不能出现分析过程中对象引用关系不断变化的情况，否则可达性分析的准确性无法保证)。

    目前主流的Java虚拟机都是准确式GC(虚拟机可以判断内存中对象的类型)，当系统停顿后，虚拟机不需要一个不漏地检查所有执行上下文和全局的引用位置，虚拟机可以知道哪些地方存放着对象引用。在HotSpot实现中，是使用一组OopMap的数据结构类达到这个目的，类加载完成后，HotSpot就把对象内什么偏移量上是什么类型的数据计算出来，在JIT编译过程中，也会在特定位置记录下栈和寄存器中哪些位置是引用，这样GC过程中可以直接得知执行上下文和全局变量位置。

  2. 安全点(SafePoint)

    通过OopMap，HotSpot可以快速准确地完成GC Roots枚举，但是可能会导致引用关系变化，或者说OopMap内容变化指令非常多，若每一条指令都生成OopMap，将会浪费大量额外空间，GC成本变高。

    HotSpot没有为每一条指令生成OopMap，而是在特定位置生成了OopMap，这些位置即安全点（SafePoint）。程序执行时并非在所有地方都停顿下来执行GC，而是只有在安全点才能暂停。安全点的选择是以程序“是否具有让程序长时间执行下去的特征”为标准进行选定的。“长时间执行”最明显特征是指令复用，例如方法调用、循环跳转、异常跳转等，所以这些具有这些功能的指令才会产生SafePoint。

    因为GC时需要程序暂停，但是只能在安全点暂停，所以GC发生时需要保证所有线程（不包括执行JNI调用的线程）都在安全点上，有两种方案：

    • 抢先式中断(Preemptive Suspension)：不需要线程执行代码主动配合，在发生GC时，首先把所有线程全部中断，若发现存在不在安全点的线程，则恢复线程，直到停顿在安全点。现在几乎没有虚拟机采用抢占式中断。
    • 主动式中断(Voluntary Suspension)：当GC需要中断线程时，不直接对线程操作，仅仅简单地设置一个标志，各个线程执行时主动去轮询这个标志，发现中断标识为真时就自己中断挂起，轮询标志的地方和安全点是重合的，另外加上创建对象需要分配内存的地方

  3. 安全区域

    Safe Point可以解决如何进入GC的问题，但是仅仅保证了程序执行时，在不太长时间内进入GC的Safe Point，若存在没有分配时间片的线程(Sleep或者Blocked状态)，线程无法响应JVM的中断请求，此事需要安全区域(Safe Region)解决问题。

    安全区域指一段代码中，引用关系不会发生变化。这个区域中任意地方开始GC都是安全的。线程执行到Safe Region中的代码时，首先标识自己已经进入安全区(Safe Region)。当JVM发生GC时，就不需要考虑标识为安全区(Safe Region)的线程了。线程离开Safe Region时，首先检查系统是否完成了根节点枚举（或者整个GC过程），若完成线程继续执行，否则必须等待收到可以安全离开的信号为止。

5. 垃圾收集器

收集算法是内存回收的方法论，垃圾收集器就是内存回收的实现。Java虚拟机规范对垃圾收集器应该如何实现并没有任何规定，所以不同的厂商、不同的虚拟机所提供的收集器有很大差别，以下是HosSpot虚拟机的垃圾收集器。

上图的7中不同垃圾收集器作用于不同分代。收集器之间有直线，说明可以配合使用。

  1. Serial收集器

    Serial收集器是最基本、发展历史最悠久的收集器，其是一个单线程的收集器，进行垃圾收集时，必须暂停其他所有的工作线程，直到收集结束，其他线程的工作由垃圾收集器暂停，所以这对用户不可见，这也对很多应用不友好。

    HotSpot虚拟机开发团队一直努力消除或者减少工作线程因为内存回收导致的停顿。从Serial到Parallel，再到Concurrent Mark Sweep(CMS)乃至Garbage First(G1)收集器。

    Serial垃圾收集器是虚拟机在Client模式下默认的新生代垃圾收集器。

    • 优点：简单高效，对象单个CPU的环境，没有多线程交互的开销

  2. ParNew收集器

    ParNew是Serial收集器的多线程版本，除了多线程进行垃圾收集外，其余行为包括Serial收集器可用的所有控制参数、收集算法、Stop The World、对象分配规则、回收策略等都与Serial一致。

    ParNew是很多运行在Server模式下的虚拟机中的首先新生代收集器，因为除了Serial之外，ParNew是唯一可以与CMS配合使用的垃圾收集器。当使用-XX:+ConcMarkSweepGC参数后，ParNew是默认垃圾收集器，-XX:+UseParNewGC指定新生代垃圾收集器。在单线程环境下，ParNew不如Serial效果好，ParNew默认开启的线程数与CPU数量相同，-XX:+ParallelGCThreads限制垃圾收集器的线程数。

    • 并行(Parallel)，指多条垃圾收集线程并行工作，此时用户线程处于停顿状态。
    • 并发(Concurrent)，用户与垃圾收集线程同时执行(可能会交替执行，也可能不在一个CPU上)

  3. Parallel Scavenge

    Parallel Scavenge是新生代垃圾收集器，同样是使用复制算法的并行多线程垃圾收集器，其与ParNew不同之处：Parallel Scavenge目标是达到一个可控制的吞吐量(Throughtput)，即CPU用于运行用户代码的时间与CPU总消耗时间的比值，即吞吐量=运行用户代码时间/运行用户代码时间+垃圾收集时间。

    停顿时间越短就越适合需要与用户交互的程序，良好的相应速度能提升用户体验，高吞吐量可以高效率地利用CPU，适合在后头运算而不需要太多交互的任务。

    Parallel Scavenge提供了两个参数用于控制吞吐量。

    • -XX:+MaxGCPauseMillis，最大垃圾收集停顿时间，大于0ms，垃圾收集器会尽量在规定时间内完成收集，GC停顿时间缩短是以缩小吞吐量和新生代空间来换取时间的。
    • -XX:+GCTimeRatio，设置吞吐量大小，0~100之间，垃圾收集时间占总时间的比率，例如：-XX:+GcTimeRatio=19，则运行最大GC时间是总时间的5%（1/1+19）。
    • -XX:+UseAdaptiveSizePoilcy，开关参数，打开后，不需要指定新生代大小(-Xmn)、Eden与Survivor的比例(-XX:+SurvivorRatio)、晋升老年代对象大小(-XX:+PretenureSizeThreshold)等细节参数，虚拟机会根据当前系统运行情况收集性能监控，动态调整以提供合适的停顿时间或者最大吞吐量，GC自适应调节策略。

    使用-XX:+UseAdaptiveSizePolicy自适应调节策略，加-XX:+MaxGCPauseMillis或-XX:+GCTimeRatio给虚拟机设定一个目标。

  4. Serial Old收集器

    Serial Old是Serial的老年代版本，单线程收集器，使用标记-整理算法，主要意义在于给Client模式下虚拟机回收老年代使用。

    • JDK5之前的版本中与Parllel Sacvenge搭配使用
    • 作为CMS收集器的后备预选，在并发收集发生Concurrent Mode Failure时使用。

  5. Parallel Old收集器

    Parallel Old是Parallel Scavenge收集器的老年代版本，使用多线程和“标记-整理算法”。其主要是与Parallel Scavenge配合使用。“吞吐量”优先收集器的组合，在一些注重吞吐量以及CPU资源敏感的场合，都可以优先考虑Parallel Scavenge和Parallel Old。

  6. CMS收集器

    CMS是一种一获取最短回收停顿时间为目标的收集器。基于标记-清除算法实现，分为4步。

    • 初始标记(CMS initial mark)，只是标记一个GC Roots能直接关联的对象，速度快，需要停顿
    • 并发标记(CMS concurrent mark)，并发标记是GC Roots Tracing的过程
    • 重新标记(CMS remark)，为了修正并发标记阶段因用户程序继续运作导致标记产生变动的那一部分对象的标记记录，比初始标记停顿时间稍长，远比并发标记时间短
    • 并发清除(CMS concurrent sweep)

    在整个过程中，最耗时的并发标记与并发清除都可以与工作线程一起工作，所以其是与工作线程并发执行的

    • CMS优点：并发收集、低停顿

    • 缺点：

      • CMS收集器对CPU资源非常敏感，会因占用CPU资源导致程序变慢，吞吐量会降低。默认启动的线程数是(CPU数量+3)/4，会随着CPU数量增大占比下降，所以当CPU低于4个时，CMS对程序影响很大
      • CMS收集器无法处理浮动垃圾，可能会出现失败导致另一次Full GC产生。所谓浮动垃圾即，CMS无法处理并发清除阶段产生的垃圾，只能留在下一次GC时清除，而且垃圾收集阶段程序会运行，所以需要预留足够的内存空间供程序运行使用，-XX:+CMSInitiatingOccupancyFraction提高触发GC的百分比，以降低内存回收次数。Concurent Mode Failure失败会导致虚拟机启动后备预案，临时启用Serial Old收集器重新进行老年代的垃圾收集，所以-XX:+CMSInitiatingOccupancyFraction太高也会导致Concurent Mode Failure失败，降低性能。

      浮动垃圾是在并发标记过程中与程序是并行的，这个过程中程序可能会产生新的垃圾(浮动垃圾)，重新标记过程只是修改并发标记过程中的不可达对象，没有办法处理浮动垃圾。

      • CMS是基于标记-清除算法实现的，会产生大量的内存碎片。-XX:+UseCMSComopacetAtFullConllection开关参数，当CMS顶不住要进行Full GC时开始内存碎片整理合并过程，但是会导致停顿时间变长。-XX:+CMSFullGCsBeforeCompaction表示设置执行多少次不压缩的Full GC后，进行一次带压缩的Full GC，默认每次都带压缩。

    • Concurent Mode Failure原因主要有两个：

      • 在年老代被用完之前不能完成对无引用对象的回收
      • 是当新空间分配请求在年老代的剩余空间中得到满足。

  7. G1收集器

    G1是一款面向服务端应用的垃圾收集器，与CMS相比优点如下：

    • 并行与并发，G1能充分利用多CPU、多核环境的硬件优势，使用多个CPU缩短停顿时间。
    • 分代收集，采用不同的方式去处理新生代、老年代的垃圾对象。
    • 空间整合，整体上看是标记-整理算法，局部看是复制算法，在GC运行后不会导致内存碎片产生，不会因为无法分配大对象无法找到连续内存空间而提前出发一次GC
    • 可预测的停顿，可以建立可预测停顿模型，可以在开发者指定时间内完成GC。

    G1收集器，Java堆的内存与其他收集器不同，其将整个堆内存划分为多个大小相等的Region，所以新生代和老年代不再是物理隔离的，是Region的集合。

    G1可以预测停顿时间，是因为可以有计划地避免在整个堆中进行垃圾回收。G1跟踪各个Region里面垃圾堆积的价值大小（回收获得的空间大小以及所需要的时间），在后台维护一个优先列表，每次根据允许的时间，优先回收价值最大的Region，同时也保证了G1在有限时间内获取尽可能高的收集效率。

    G1划分Region后，Region仍不是孤立的，因为对象分配到Region后，其可以被其他Region内的对象引用。G1中Region之间的对象引用以及其他收集器中新生代和老年代之间的对象引用，虚拟机使用Remembered Set避免全堆扫描。在G1中每个Region都有一个Remembered Set。当虚拟机发现需要对Reference类型的数据写操作时，会触发一个Write Barrier暂时中断写操作，检查Reference引用的对象是否处于不同Region中，若是则通过CardTable把相关信息记录到被引用对象所属的Regioin的Remebered Set中，内存回收时在GC Roots中加入Remembered Set即可保证不对全堆扫描也不会遗漏。

    不包括Remembered Set操作，G1收集器分为以下几个步骤：

    • 初始标记(Initial Marking)，标记GC Roots能直接关联到的对象，并写修改TAMS(Next Top at Mark Start)的值，让下一阶段用户程序并发运行时，能正确在可用的Region中创建对象。需要停顿线程，时间很短
    • 并发标记(Concurrent Marking)，从GC Roots开始对堆中对象进行可达性分析，找出存活的对象，耗时较长，可以与程序并发运行
    • 最终标记(Final Marking)，是为了修正并发标记期间因为用户程序继续运行导致标记产生变动的记录，变动记录在Remebered Set Logs里面，最终标记会把Remebered Set Logs的数据合并到Remebered Set中，需要停顿，但是可以并行执行
    • 筛选回收(Live Data Counting And Evacuation)，对各个Region的回收价值和成本进行排序，根据用户期望指定回收计划，可以并发执行

  8. 理解GC日志

    1. 33.125和100.667表示GC发生的时间，从Java虚拟机启动开始计数。
    2. [GC和[Full GC是指垃圾收集的停顿类型，Full说明是带有停顿的过程。System.gc()触发的收集，显示为[Full GC(System)。
    3. [DefNew、[Trnured、[Perm表示GC发生的区域
    4. 3324K->152K(3712K)，是指GC前该区域内存已使用量 -> GC后该区域内控使用量(该区域内存总量)。方括号之外的3324K->152K(11904K)表示GC前该Java堆已使用量 -> GC后该堆使用量(Java堆总容量)

6. 内存分配与回收策略

内存自动管理技术可以归结为自动化地解决了两个问题：

  • 对象分配内存，各种GC收集器算法及实现，枚举根节点、安全点、安全区域等HotSpot算法等等
  • 回收对象内存，堆上分配内存，主要分配在Eden上，如果启动了本地线程缓冲，将按线程优先在TLAB上分配。分配规则不是百分百确定，而是当前使用的垃圾收集器组合以及虚拟机参数设置的，以下是主要的内存分配规则
    • 对象优先在Eden分配
    • 大对象直接进入老年代
    • 长期存活的对象进入老年代
    • 动态对象年龄判定
    • 空间分配担保
  • 以下举例均以Parallel Scavenge + Parallel Old垃圾收集器组合为例

  1. 对象优先在Eden分配

    对象优先在Eden分配，当Eden区域没有足够空间时，虚拟机将发起一次Minor GC。-XX:+PringGCDetails，用于打印在GC发生垃圾收集行为时打印内存回收日志，在进程退出时输出当前内存区域分配情况，实际应用中使用日志工具进行分析。

    • 新生代GC(Minor GC)，指发生在新生代的垃圾收集工作，因为Java对象大多生存时间短，所以Minor GC非常频繁，回收速度很快。
    • 老年代GC(Major GC/Full GC)，发生在老年代的GC，Full GC经常会伴有Minor GC，但并不绝对，且Full GC速度比Minor GC慢10倍以上。

    执行结果如下:

    byte[] allocation4 = new byte[4 * _1MB];会触发一次Minor GC，因为超过了Eden分配空间，Minor GC后6906K->973K(9216K)，说明GC后3个对象分配到了老生代(对象超过Survivor空间大小，所以会通过内存分配担保提前转移到老年代去)。

  2. 大对象直接进入老年代

    大对象指需要大量连续内存空间的Java对象，最典型的就是很长的字符串以及数组，实际开发中应该尽量避免使用大对象，容易导致还有不少内存空间而触发垃圾收集以获取足够连续内存空间。-XX:PretenureSizeThreshold参数，用于设置超过指定大小后直接分配到老年代分配，这是为了避免在Eden区与Survivor之间发生大量内存复制。

  3. 长期存活对象进入老年代

    分代回收对象管理内存，那么内存回收哦时必须知道哪些对象应该放在新生代，哪些在老年代。虚拟机给每个对象定义了一个对象年龄(Age)计数器。Eden -> Surivivor对象年龄设置为1，且每经过一次Minor GC，年龄加1,当达到一定次数后将会晋升到老年代。-XX:MaxTenuringThreshold设置年龄阈值。

  4. 对象年龄判定

    为了更好适应不同程序的内存状况，虚拟机并不是永远要求对象年龄必须达到阈值(MaxTenuringThreshold)才会晋升到老年代。当Survivor空间中相同年龄所有对象大小的和大于Survivor空间的一半，年龄大于等于该年龄的对象就可以直接晋升到老年代。

  5. 空间分配担保

    发生Minor GC之前，虚拟机会先检查老年代最大可用的连续空间是否大于新生代所有对象总空间，成立则Minor GC是安全的，不成立则虚拟机会查看-XX:HandlePromotionFailure设置的值是否允许担保失败。若允许则会继续判断老年代最大可哟连续空间是否大于历次晋升到老年代对象的平均大小，若大于则尝试进行一次Minor GC（可能会有风险），若小于或者-XX:HandlePromotionFailure设置为不允许冒险，则改为进行一次Full GC。

7. 总结

介绍了垃圾收集器特点以及运作原理，验证了Java虚拟机分配内存以及回收的主要规则。内存回收与垃圾收集器很多时候是影响系统性能、并发能力的主要因素之一。
