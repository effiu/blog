Java 8 函数式编程

Posted on 2020-03-21 16:11 Edited on 2020-03-29 16:02 In Java Views: Valine: Reading time ≈ 17 mins.

随着时代的发展，越来越需要更复杂的应用程序。高效运行时编译器的Java虚拟机，使得我们可以将更多精力放在编写干净、易于维护的代码上。

现在计算机发展方向是多核CPU而不是高频率，但是多核情况下涉及锁的编程算法不但容易出错， 而且耗费时间。 人们开发了java.util.concurrent包和很多第三方类库， 试图将并发抽象化， 帮助程序员写出在多核 CPU 上运行良好的程序。 很可惜， 到目前为止， 我们的成果还远远不够。

为了能够使用 Java 8 编写复杂的集合处理算法，只需要修改一个方法，就能够让代码在多核CPU上高效运行。为了编写类处理批量数据的并行类库，增加了Lambda表达式。

面向对象编程是对数据进行抽象，函数式编程是对行为进行抽象。现实世界中，数据和行为是并存的。

函数式编程的核心是：使用不可变值和函数，函数对一个值进行处理，映射成另一个值。

2. 辨别Lambda表达式

Lambda 表达式依赖于上下文环境，是由编译器推断出的。final String[] array = { "hello", "world" };中数组的类型就是根据上下文推断出来的。null本身并没有类型，而是只有String str = null;将null赋值给一个变量后才能知道其类型。

3. 引用值，而不是变量

在匿名内部类中，需要引用其方法里的变量时，需要将变量声明未final，即不能重新赋值。Java 8 中放松了这个限制，不在需要显式声明为final变量，但是其变量本身需要是final的(即只能被赋值一次，不能被重新赋值)。

在匿名内部类中使用方法里的变量时，若不声明为final，则会出现问题；但是匿名内部类引用外部类的成员变量则不需要声明为final。这是与Java内存区域的堆栈有关的。

匿名内部类在Javac编译后，JVM会为匿名内部类单独生成一个class文件，匿名内部类引用的变量会被copy到内部类的环境中。而如果外部类的方法变量被修改后，会导致内部值与外部值不一致，从而产生意料之外的情况。而堆内存储的是Class文件信息、常量池、对象以及对象包含的类的成员变量等信息。

栈是Java方法执行的内存模型，每调用一个方法就会为每个方法生成一个栈帧。任何操作都是通过入栈和出栈完成的，栈内存储的是局部变量表、操作数栈、动态链接、方法出口等信息。当匿名内部类引用的变量是方法内部变量时，其copy的是调用方方法栈帧中的值，copy后原来栈帧变为不可见；当匿名内部类引用的是外部类的成员变量时，调用方方法栈帧中保存的是指向堆的引用，其实际上是不会变化的(栈帧内指向堆的指针不会变化)，不会出现意料之外的情况，所以不需要声明为final。

5. 类型推断

某些情况下，用户需要手动指明类型，有些时候省略类型信息可以减少干扰，更易阅读。Java 7引入了类型推断，List<Long> list = new ArrayList<>();中，在new对象时没有明确指明类型信息，而是使用菱形操作符，编译器可以自己推断出其实际类型。

Lambda表达式中的类型推断是Java 7引入的目标类型推断的扩展。我们将一个未指明类型的构造函数直接传递给一个方法，也可以根据方法签名推断类型。例如:private void** useHashmap(Map<String, String> values);,useHashmap(new HashMap<>())调用,就可以根据方法签名推断类型。Java 7中可以根据方法签名推断构造函数的泛型类型，而Java 8则可以省略Lambda表达式中的所有参数类型, javac根据Lambda表达式上下文信息就能推断出参数的正确类型。虽然程序依然要经过类型检查来保证运行的正确性, 但是已经不再需要显式声明。

通常在Java中调用一个方法，计算机会立即执行操作，但是Stream却并不是这样。

判断一个操作是惰性求值还是及早求值很简单：只需看它的返回值。返回值为Stream对象，那么则是惰性求值；若返回另一个值或者空，则是属于及早求值方法。其理性操作是，使用惰性方法形成一个操作链，最后用一个及早求值的操作返回一个想要的结果。

3. 类库

即使我们不需要编写像Stream这样的Lambda表达式，也需要学会如何使用Lambda表达式。Java 8中的另一个变化是引入了默认方法和接口的静态方法。

1. 基本类型

Java中存在基本类型，而其相伴类型是装箱类型。基本类型内建在语言和运行环境中，是基本的程序构建模块；而装箱类型是普通的Java类，是对基本类型的一种封装。

Java的泛型是基于对泛型类型参数的擦除，其是Object对象的实例，所以只有装箱类型才能作为泛型参数。但是装箱类型是对象，在内存中存在额外开销，例如int在内存中占用4个字节，Integer对象却占用16个字节，甚至整形数组对象中，由于元素是内存中的指针，指向Java堆中的对象，同样大小的数组，Integer[]比int[]多占用6倍内存。

装箱和拆箱都需要额外的开销。对于需要大量数值运算的算法来说，装箱和拆箱的计算开销，以及装箱类型占用的内存开销，会明显减缓程序的运行速度。

为了减少上述内存开销，Stream类的某些方法对基本类型和装箱类型做了区分。如下图:

目前Java 8中仅仅对整形、长整型和双浮点数做了特殊处理，特殊处理后的系统性能明显提升。上图中的命名规范以及返回类型，可以明白其使用场景。

2. 重载解析

Java中可以重载方法，造成多个方法有相同的方法名，但是方法签名不一样。这给类型推断带来了问题，系统可能会推断出多种类型，这时javac会挑出最具体的类型。当参数为Lambda表达式类型时，同样如此。

overloadedMethod((x, y) -> x + y);会被推断为IntegerBinaryOperator，因IntegerBinaryOperator是BinaryOperator<Integer>的子类。

当存在多个重载方法时，哪个是最具体的类型可能并不准确，编译器会在此时提示编译错误，此时可以将参数强制转化为你所期望的类型。

Lambda表达式作为参数时，其类型由它的目标类型推导得出，推导过程遵循如下规则:

  • 如果只有一个可能的目标类型，由相应函数接口里的参数类型推导得出;
  • 如果有多个可能的目标类型，由最具体的类型推导得出;
  • 如果有多个可能的目标类型且最具体的类型不明确，则需人为指定类型。

3. @FunctionalInterface

每个用作函数接口的接口都应该添加 @FunctionalInterface注释。每个用作函数接口的接口都应该添加该注释。Java中有一些只有一个方法的接口，但其并不是为了Lambda表达式实现的。例如：有些对象内部可能保存着某种状态，使用带有一个方法的接口可能纯属巧合。java.lang.Comparable 和java.io.Closeable就属于这样的情况。

与Closeable等接口不同，为了提高Lambda对象的可操作性而引入了各种接口，都需要Lambda表达式实现。其意义在于将代码块进行打包。都使用了@FunctionInterface注释。其会强制javac检查一个接口是否符合函数接口的标准。若给该接口添加个枚举类型、类或者接口不仅一个抽象方法，javac就会报错。

4. 二进制兼容性

Java 8对API最大的改进是集合类。虽然Java一直在持续演进，但是一直保持着向后二进制兼容。具体就是Java 1到Java 7编译的类库或者应用，可以在Java 8上运行。

当然也会有错误出现，但是二进制兼容性一直是Java的优势。除非引入新的关键字，如enum，达成源代码向后兼容也不是没有可能实现。可以保证，只要是Java 1到Java 7写出的代码，在Java 8中依然可以编译通过。

实际，在Java 8中修改了集合类类库后，已经很难实现。Java 8中为Collection接口增加stream方法，这意味着所有实现了Collection接口的类都必须增加这个新方法。对于核心类库来说，增加stream方法，就可以解决。但是对于JDK之外的类(例如继承了ArrayList的类)，也需要实现stream方法，否则无法通过编译。

为了避免上述问题，在Java 8中添加了新的语言特性：默认方法。

5. 默认方法

Collection接口中增加了新的stream方法，若其子类中没有实现stream方法，就会默认使用Collection的stream方法。Iterator中也增加了一个默认方法:forEach，与for循环类似，允许用户使用一个Lambda表达式作为循环体。

Java 8新增了关键子default,这个关键字告诉javac用户真正需要的是为接口添加一个新方法。default方法在继承规则和普通方法也有区别，与类不同，接口没有成员变量，因此默认方法只能通过调用子类的方法来修改子类本身。

上述代码说明：类中重写的方法优先级高于接口中定义的默认方法。这是由增加default方法的目的决定的，其主要是为了在接口上向后兼容，这可以简化很多继承问题。假设存在一个MyCustomList，该类中有一个addAll方法，如果新的List接口也增加了一个默认方法addAll，该方法将对列表的操作代理到add方法。如果类中重写的方法没有默认方法的优先级高，那么就会破坏已有的实现。

6. 多重继承

上述代码中，若C1不重写m方法，会编译报错inherits unrelated defaults for m(),需要我们在类中重写m()

对于default默认方法的工作原理，有如下三个规则：

  • 类胜于接口，若在继承链中有方法或抽象方法声明，那么就忽略接口中定义的方法。
  • 子类胜于父类，若一个接口继承了另一个接口，且两个接口中都定义了一个默认方法，那么子类中定义的方法胜出
  • 若上述两个规则仍然不能确定，子类就需要实现该方法或者声明为抽象方法。

7. 接口的静态方法

  • of()
  • terate()
  • concat()

8. Optional

Optional是为核心类库设计的一个数据类型，用来替换null值。使用null代表值不存在的最大问题在于 NullPointerException。而Optional可以完美解决这个问题。

4. 高级集合类和收集器

1. 方法引用

在Lambda中，方法调用a ->a.toString()。Java 8提供了一个简单的语法：方法引用，用来帮助程序员重用已有方法，写法：Classname::methodName，其是一种与Lambda表达式等价的结构。构造函数同样有缩写形式：Classname::new，在选对函数接口的情况下，还支持多个构造参数。数组写法：String[]::new。

2. 元素顺序

流中的元素顺序，依赖于集3合和对流的操作。例如，List是有序的，List创建的流是有序的；HashSet创建的流就是无序的，HashSet本身无序。

流的目的不仅仅是在集合类之间做转换，而且同时提供了一组处理数据的通用操作。一些中间操作会产生顺序，比如对值做映射，映射后的值是有序的。

一些流在有序的流上开销很大，调用unordered方法消除这种顺序可以解决问题。大多数操作都是在有序流上面效率更高，如filter、reduce、map等。无序会带来一些意想不到的结果，并行流中forEach方法并不能保证元素执行有序，若要保证元素按序处理，则使用forEachOrdered方法。

  • forEach方法，用于快速迭代

5. 数据并行化

从之前的外部迭代到Java 8的内部迭代，使编写简洁的代码容易了很多，另一个好处则是程序员不需要手动控制迭代过程，甚至只需要轻微的改动，就可以使迭代并行。那么为什么需要并行化以及什么时候并行会带来性能的提升呢？

1. 并行和并发

并行和并发是两个概念，并行是两个任务在同一时间发生，例如运行在多核CPU上；并发则是一个程序要运行多个任务，同一个CPU上分配了不同的时间片运行不同任务。如下图:

并行是为了缩短任务执行时间，将一个任务分解成几部分，然后并行执行。与串行任务量一样，只不过是以更多CPU执行该任务，以空间换时间，实际上并行执行时，CPU承载的工作量更大。

数据并行化是指将数据分成块，为每块数据单元分配单独的处理单元。当需要在大量数据上执行同样的操作时，数据并行化很管用。其将问题分解为可以在多块数据上求解的形式，然后对每块数据执行运算，最后将各数据块上得到的结果汇总，从而得到最终答案。

任务并行化与数据并行化的区别：任务并行化中，线程不同，工作不同。

2. 并行化流操作

Java 8中并行化流只需要改变一个方法调用。stream.parallel()或者直接调用parallelStream()获取并行流。

然而实际运算中，并不是并行一定比串行快，而是当需要计算的任务达到一定数量规模后，并行效率才会优于串行。而这个规模是最则依赖任务复杂度。输入流的大小并不是决定并行化是否会带来速度提升的唯一因素，速度性能还会受到编写代码的方式和核的数量的影响。

3. 性能

影响并行流性能的主要因素有5个：

  • 数据大小：输入的数据量大小影响并行对性能的提升，将问题分解之后进行并行处理，再将结果合并会带来额外的开销。因此只有数据足够大、每个数据管道花费的时间足够多时，并行化才有意义。
  • 源数据结构：每个管道的操作都基于一些初始数据源，通常是集合。将不同的数据源分割相对容易，这里的开销影响了在管道中并行处理数据时到底能带来多少性能上的提升。
  • 装箱：处理基本类型比处理装箱类型快
  • 核的数量：若是单核CPU，则不需要并行化。拥有的核数越多，获得的潜在性能提升的幅度就越大。实践中，核的数量不单指机器上的核数，更是指运行时机器能使用多少核，以及若使用多核是否会影响其他进程或者线程。
  • 单元处理开销：执行并行花费时间与分解合并操作开销之间的战争。花在流中每个元素身上的时间越长，并行操作带来的性能提升越明显。

并行流的底层是fork/join。fork递归式分解问题，然后每段并行执行，最终由join合并结果，得到最终结果。由于任务需要分解，那么初始数据源的特性变得很重要，其影响了分解的性能。直观上看，能重复将数据结构对半分解的难易程度，决定了分解操作的快慢。

根据分解任务的性能，将核心类库提供的通用数据结构分成以下3种情况：

  • 性能好，例如:ArrayList、数组或IntStream.range，这些数据结构支持随机读取，其可以轻而易举地被任意分解
  • 性能一般，HashSet、TreeSet，不易被公平分解，但是是可分解的
  • 性能差，难于分解，甚至需要遍历，复杂度为$O(n)$。例如：LinkedList、Streams.itreate和BufferedReader.lines。

理想情况下，流框架将问题分为不同小块，就可以单独处理每一小块，线程之间不需要进一步通信。其实际上有的需要维护状态，所以可能会有额外的开销和限制。

无状态的性能更好，例如map、flatMap、filter等。有状态的sorted、distince、limit等

4. 并行化数组操作

Java 8还引入了一些针对数组的并行操作，脱离流框架也可以使用Lambda表达式。与流框架的操作一样，是针对数组的并行化操作。

java.util.Arrays类中：

  • public static <T> void parallelPrefix(T[] array, BinaryOperator<T> op)，任意给定一个函数，对数组组做计算
  • parallelSetAll，使用Lambda表达式更新数组元素
  • parallelSort，并行化对数组元素排序

6. 其他

我们有时候在流中需要打印下运行信息，但是有一些过程是惰性求值的，所以很难准确打印每一步的信息。可以使用peek方法，既可以查看每个值，同时可以继续操作流。
