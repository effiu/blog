Java Web中文编码

Posted on 2020-03-26 01:09 Edited on 2020-03-29 16:00 In Java Views: Valine: Reading time ≈ 8 mins.

编码问题一直是开发人员经常遇到的问题，尤其是Java，因为Java是跨平台语音，而不同平台之间的编码切换非常多。对于国内的程序员更甚，因为还要转换成汉语。

  • 编码格式的区别
  • 要编码的场景
  • 中文编码问题的原因以及如何解决
  • Java Web开发中可能会存在编码的几个地方
  • 一个HTTP请求怎么控制编码格式

1. 常见的编码格式

byte是计算机中的一个基本存储单元，其无法一一表示人类的语言，所以必须要经过翻译，才能让计算机理解我们的语言，这个编译过程就是编码。

计算机提供了多种编码方式，ASCII、ISO-8859-1、GB2312、GBK、UTF-8、UTF-16等。其中支持汉子的有GB2312、GBK、UTF-8、UTF-16，按照这些编码规则就可以让计算机正确表示我们的字符。

  1. ASCII码

    ASCII码共有128个，用1个字节的低7位表示。0～31表示控制字符，如换行、回车、删除等；32～127是打印字符，如0～9，a～z，A~Z等

  2. ISO-8859-1

    ISO组织在ASCII的基础上扩展了ASCII码，其涵盖了大部分西欧字符。ISO-8859-1是单字节编码，共256个。

  3. GB2312

    GB2312是双字节编码，总编码范围是A1～F7。A1～A9是符号区，共包含682个字符，B0～F7是汉子区，共包含6763个汉字

  4. GBK

    扩展了GB2312，加入了更多的汉字。其编码范围是8140～FEFE(去掉XX7F)，共23940个码位，能表示21003个汉字，其兼容GB2312

  5. UTF-16

    UTF-16具体定义了Unicode字符在计算机中的存取方法，UTF-16用两个字节表示Unicode的转化格式，采用定长的方法，即任何字符都占用两个字节

  6. UTF-8

    UTF-16用两个字节的定长方法表示字符，虽然简单、方便，但是很多字符用一个字节就可以表示，这大大浪费了存储空间，这在目前带宽有限的情况下会增大网络传输的流量。UTF-8采用了变长技术，每个编码区域有不同的编码长度，不同类型的字符可以由1～4个字节组成。编码规则如下：

    • 若是1个字节，最高位为0，则表示这是一个ASCII字符(00~7F)。UTF-8兼容ASCII码
    • 若是1个字节，以11开头，则连续的1表示这个字符的字节数
    • 若是1个字节，以10开头，表示非首字节，则需要向前查找才能得到当前字符的首字节

2. Java中需要编码的场景

1. I/O操作中的编码

涉及到编码的地方都是在字节与字符之间的转换发生的，所以主要场景是IO上，包括磁盘IO和网络IO。

Reader类是Java IO中读字符的父类，而InputStream是读字节的父类，InputStreamReader是字节与字符之间的桥梁，其负责处理读取字节到字符的转换，对于具体字节到字符的解码实现，其又委托StreamDecoder完成，在解码过程中必须由用户指定charset编码格式。默认使用系统本地的字符集，中文环境是GBK。下面是编码和解码的例子：

3. Java中如何编解码

首先根据Charset.forName(String charName)找到Charset类，然后根据Charset创建CharsetEncoder，每个编码方式都有自己的CharsetEncoder实现。然后调用CharsetEncoder.encode类对字符串进行编码。

也是通过charsetName找到Charset类，然后根据编码集生成CharsetEncoder，CharsetEncoder是所有字符编码的父类，然后调用CharsetEncoder.encode进行编码。

对于中文字符，GB2312、GBK、UTF-8、UTF-16都可以使用，其中GB2312与GBK规则类似，但是GBK范围更大，它能处理所有汉字字符，所以应该尽量选择GBK。UTF-16与UTF-8则是两种编码规则，UTF-16编码效率高，从字符到字节相互转换简单，进行字符串操作更好，适合在本地磁盘和内存之间使用，可以进行字符和字节之间的快速切换，但是UTF-16不适合在网络之间传输，因为网络传输容易损坏字节流，一旦发生将很难恢复，所以相比UTF-8更适合网络传输。UTF-8对ASCII字符采用单字节存储，另外单个字符损坏也不会影响后续的其他字符，编码效率介于GBK和UTF-16之间。

4. Java Web中涉及的编码

对于中文来说，有IO的地方就会有编码，IO操作会发生编码转换，而大部分乱码都是网络IO引起的。我们需要考虑两个问题：

  1. 一段文本的实际大小应该如何计算，例如，把整型数字1234567作为字符串处理，采用UTF-8将会占用7个字节(ASCII码范围)，采用UTF-16则会占用14个字节，但是若把其作为int存储则只会占用4个字节；我们压缩Cookie时，会选择不同的压缩算法，发现压缩后的字符数是减少了，但是字节数并没有减少。所谓的压缩只是将多个单字节通过编码转换为一个多字节。
  2. 当我们计算机中的某个文本编辑器里输入汉字时，其到底是如何表示的。计算机中所有的信息都是以0和1表示的，那么一个汉字是如何用0和1表示呢？Java中一个char是16个bit，2个字节，当汉字用char表示后，在内存中会占用相当于4个字节的空间。

理解上面两个问题后，那么用户从一个浏览器发起一个HTTP请求后，需要存在编码的地方是URL、Cookie、Parameter。服务接受到请求后解析HTTP请求，其中URI、Cookie和POST表单都需要解码，甚至服务器端可能还需要读取数据库中的数据也会发生解码等等。当服务器处理完请求的数据后，需要将这写数据再编码，通过Socket发送到用户请求的浏览器里面，再经过浏览器编码成文本，这个过程最少发生2次编码、2次解码，若发生数据库行为，则会更多。

上图中可以发现，URI编码与参数编码不一致，这是GET请求。浏览器编码URL是将非ASCII码按照某种编码格式编码成16进制数字后将每个16进制表示的字节前加上’%’。对URL的URI部分进行编码的字符集是在connector的<Connector URIEncoding="UTF-8"/>中定义的，若未定义，那么将以默认编码ISO-8859-1解析。

对于GET请求参数的解析：GET请求和POST请求的表单参数都是作为Parameter保存的，都通过request.getParameter获取参数值。其解码过程发生在第一次调用request.getParameter时，其在第一次被调用时将会调用org.apache.catalina.connector.Request的parseParameter方法，这个方法将会对GET和POSt方式传递的参数进行解码，但是它们的解码字符集可能不一样。GET请求的解码字符集是通过HTTP的Header传到服务端的，其与对URI编码的字符集不一致。对参数的编码字符集要么从Header中得到，要么是默认的ISO-8859-1。若要使用Header中ContentType中定义的编码，要将connector中的<Connector URIEncoding="UTF-8" useBodyEncodingForURI="true"/>。

URL编码解码是比较复杂的，所以我们尽量在URL中使用非ASCII码字符，否则可能会出现乱码问题。服务端也要设置<Connector URIEncoding="UTF-8" useBodyEncodingForURI="true"/>

2. HTTP Header编解码

当客户端发起一个HTTP请求时，除URL外，还可以能header中携带信息，如Cookie等，所以也会存在编码问题。对Header的编码是在第一次调用request.getHeader时进行的，其是调用org.apache.tomcat.util.buf.MessageBytes.toString方法，使用ISO-8859-1将byte编码为char的，没有办法设置Header的编码方式，所以若Header中出现非ASCII编码方式，则会出现乱码。

若一定要传递中文，则需要使用URLEncode.encode将中文字符编码成ASCII，在加入到Header中，然后由服务器进行URLDecode。

3. POST表单的编解码

POST表单提交的参数的解码也是在第一次调用request.getParameter()时发生的，POST参数是通过body传给服务端的，我们在浏览器中提交代码时，首先根据contentType定义的charset将参数编码，提交到服务端后同样会用ContentType中的charset解码。我们也可以通过request.setCharacterEncoding()设置编码格式。但是一定要在request.getParameter()之前设置，否则不起作用。若contentType没有设置编码方式，那么表单将会按照系统默认的编码方式解析。

  • 查看解析post参数的源码

4. HTTP BODY的编解码

当用户请求的资源成功获取后，将会通过response返回给客户端浏览器。这个过程先要经过服务器编码，再到浏览器解码。编解码字符集可以通过response.setCharacterEncoding()来设置，其也会服务request.setCharacterEncoding的值，会通过response的header返回给客户端，浏览器也是根据header中的编码方式进行解码，若header中没有设置编码方式，那么浏览器将会根据HTML中的<meta HTTP-equiv="Content-Type" content="text/html; charset=GBK"/>中的charset来解码，若仍没有定义，那么将会使用浏览器默认编码方式解码。优先级：

  • response header中的编码方式
  • HTML中规定的编码方式
  • 浏览器默认的编码方式

5. 其他需要编码的地方

  • XML文件中，<?xml version="1.0" encoding="UTF-8">
  • JSP文件中，<%@page contentType="text/html;charset=UTF-8"%>
  • JS中
    • escape()，将ASCII字母、数字、标点符号之外的其他字符转化为Unicode编码值，且在编码值前加%u
    • encodeURI()，用来对URL进行编码的函数，将整个URL中的字符除部分ASCII码外的字符进行UTF-8编码，在每个码值前加上%
    • encodingURIComponent()，支持更少的ASCII码，常用于将一个url作为参数放到另一个url中。

5. 乱码问题分析

Java中涉及到IO的地方(网络IO和磁盘IO)，绝大部份都涉及到编码。那么遇到乱码问题如何处理呢？乱码出现在char到byte或者byte到char中编码和解码的字符集不一致导致的，甚至一次操作设计了多次编解码，更很难确定哪个环节出现问题。

1. 中文字符改变

“this is a 测试”对应的字符集如下：

t h i s i s a 测 试
十进制 116 104 105 115 32 105 115 32 97 32 27979 35797
十六进制 0074 0068 0069 0073 0020 0069 0073 0020 0061 0020 6d4b 8bd5

而测试两个字对应的GBK编码，测B2E2、试CAD4。但是Uncode和GBK字符集都包含“测试”两个字，只是位置不同，所以GBK把“测试”解码成了其他字。

3. 汉字变成问号

将中文符号经过ISO-8859-1编码后，所有字符变成了？，这是因为ISO-8859-1编码时，遇到不在码值范围的统一用3f即**?**表示。

6. 总结

总结了几种常见的编码方式，介绍了支持中文的编码方式，比较了说明场景；其次介绍了Java中涉及到的编码问题，以及如果实现编码的支持，以网络IO为例，说明了HTTP请求过程中发生编码与解码的地方，最后说明了平时遇到的发生乱码问题的原因。

要解决中文编码问题，首先要清楚哪些地方会引起字符到字节的编码，以及从字节到字符的编码。最常见的就是发生磁盘IO或者网络IO的时候；应该清楚发生编码行为的地方的原理或者系统是如何控制编码的。
