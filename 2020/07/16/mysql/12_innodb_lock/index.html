Mysql InnoDB 何时加锁的整理

1. 准备工作

  • 查看事务:SELECT * FROM information_schema.INNODB_TRX;
  • 查看锁:SELECT * FROM information_schema.INNODB_LOCKS;
  • 表:team_number

2. ReadView

1. Repeated Read

在Repeated Read下，只会在创建事务时创建ReadView，默认的事务隔离级别

  1. 会话1
  1. 会话2
  1. 执行
    1. 执行会话1的查询，但不提交事务
    2. 执行会话2不提交事务，之后执行会话1的查询。
    3. 会话2事务提交，执行会话1的查询
    4. 比较上述3个查询结果

2. Read Committed

在Read Committed下，每次查询都会创建一次ReadView

  1. 会话1

  1. 会话2同上

  2. 执行

    1. 执行会话1的查询但不提交事务
    2. 执行会话2但不提交事务，执行会话1的查询
    3. 提交会话2的事务，执行会话1的查询
    4. 比较结果
    5. 提交会话1的事务

3. 锁

1. MVCC的缺陷

在REPEATABLE READ隔离级别下，默认不加锁，只在第一次执行SELECT时生产一个ReadView，把脏读、不可重复读、幻读都解决了，但是其存在缺陷。

  1. 事务1
  1. 事务2

MVCC会在第一步开启事务1时，创建一个ReadView，之后创建事务2并提交。ReadView并不能阻止T1执行Update或者Delete对事务2插入的数据做改动，这就会使新事务的trx_id变成了事务1的事务Id，之后Select该记录，就会变得可见，所以MVCC并不能完全解决幻读问题。

2. 多粒度锁

  • 行锁和表锁
  • S锁( Shared Lock)和X锁(Exclusive Lock)
  • 意向锁(Intention)

IX锁、IS锁是表级锁，是为了在之后加表级别的X锁和S锁时可以快速判断表中记录是否存在锁(防止遍历表记录判断是否存在锁)

兼容性 X IX S IS
X 不兼容 不兼容 不兼容 不兼容
IX 不兼容 兼容 不兼容 兼容
S 不兼容 不兼容 兼容 兼容
IS 不兼容 兼容 兼容 兼容

3. 锁的分类

  1. 表锁

    1. 表级别的X锁和S锁，DML语言不会加表锁;DDL语言则会锁表
    2. 表级别的IS锁和IX锁，在对表中的记录加S锁和X锁时，要先对表加IS锁、IX锁
    • 表级别的AUTO-INC锁

  2. 行锁

    1. 记录锁(LOCK_REC_NOT_GAP)，lock_type=RECORD即仅仅把一条记录锁上。行锁区分S锁和X锁。如下图是给id=3的记录加记录锁。

    2. 间隙锁(LOCK_GAP)，用来解决MVVC的缺陷，当事务读取的记录不存在时，无法给记录加锁，需要使用Gap Lock，下图中给id=6的记录加了间隙锁，这意味着在事务期间(5,6)区间内不允许插入记录。

    忽略id为主键，可以假设在name(张帅,匿名)区间内加间隙锁。若在该区间内插入记录,则会阻塞，直到锁释放。gap锁本质是为了防止插入新纪录，gap锁并不会限制对记录加其他锁，这仅仅是为了解决幻读问题

    1. Next Key Locks,当我们既想锁住某条记录，又想阻止其他事务在该记录前面插入数据时，就需要使用LOCK_ORDINARY。其本质是LOCK_REC_NOT_GAP和LOCK_GAP的合体。

    2. Insert Intention Locks,当我们要插入一条新纪录时，需要判断一下插入位置是否被别的事务加了LOCK_GAP锁，若存在锁，则事务等待，直到拥有LOCK_GAP锁的事务提交(释放锁)。但是插入记录的事务在等待时也需要在内存中生成一个锁结构，表明有事务想在间隙中插入新纪录。官方命名为:LOCK_INSERT_INTENTION，即插入意向锁。

4. 语句加锁分析

SQL语句加锁本身是一个伪命题，因为一个语句加锁有很多制约条件，例如:

  • 事务隔离级别
  • 执行时使用的索引
  • 查询条件
  • 具体执行的语句类型

加锁是为了解决并发事务执行过程中引起的脏写、脏读、不可重复读、幻读。其中MVCC算是一种解决脏读、不可重复读、幻读的一种方案。加锁是为了解决上述问题不同场景，需要加的锁不一样，不能为了加锁而加锁。

正常工作中，SQL语句大致分为3类，普通SELECT语句、锁定读的语句(UPDATE、DELETE等)、INSERT语句。

采用加锁的方式解决读-写问题时，不可重复读、脏读都不会发生，因为所有行为都要排队进行。

1. 普通SELECT语句

  • READ UNCOMMITTED隔离级别下，不加锁，直接读取记录的最新版本，可能发生脏读、不可重复读和幻读问题。
  • READ COMMITTED隔离级别下，不加锁，在每次执行普通的SELECT语句时都会生成一个ReadView，这样解决了脏读问题，但没有解决不可重复读和幻读问题。
  • REPEATABLE READ隔离级别下，不加锁，只在第一次执行普通的SELECT语句时生成一个ReadView，这样把脏读、不可重复读和幻读问题都解决了，但是MVVC存在缺陷。
  • SERIALIZABLE隔离级别下，需要分为两种情况讨论：
    • 禁用自动提交时，普通的SELECT语句会被转为SELECT ... LOCK IN SHARE MODE这样的语句，也就是在读取记录前需要先获得记录的S锁，具体的加锁情况和REPEATABLE READ隔离级别下一样。
    • 启用自动提交时，普通的SELECT语句并不加锁，只是利用MVCC来生成一个ReadView去读取记录。因为启用自动提交意味着一个事务中只包含一条语句，一条语句也就没有不可重复读、幻读这样的问题。

2. 锁定读

  • SELECT ... LOCK IN SHARE MODE
  • SELECT ... FOR UPDATE;
  • UPDATE ...
  • DELETE ...

UPDATE和DELETE因为在执行过程中需要先定位被改动的记录并给记录加锁，也需要先读取

锁定读语句，在READ UNCOMMITTED/READ COMMITTED隔离级别下是一致的。

使用主键等值查询

  • SELECT * FROM team_number WHERE id = 6 LOCK IN SHARE MODE; 为id=5的记录加S锁。

  • SELECT * FROM team_number WHERE id = 6 FOR UPDATE;为id=5的记录加X锁。

  • UPDATE ...... 为记录加锁，若锁不涉及二级索引则，其只加RECORD类型的X锁。若UPDATE语句更新列包含二级索引，例如UPDATE team_number SET name = '佚名' WHERE id = 6;。这时会先对id = 6的聚簇索引加X锁,然后对id = 6对应的二级索引的记录加X锁

  • DELETE ......对聚簇索引和所有的二级索引中对应的记录做DELETE操作。

使用主键进行范围查询

  • SELECT * FROM team_number WHERE id <= 5 LOCK IN SHARE MODE;，S锁

  1. 先到聚簇索引中定位到满足id <= 5的第一条记录，也就是id值为1的记录，然后为其加锁。

    1. 判断一下该记录是否符合索引条件下推中的条件。
    2. 该记录是否符合范围查询的边界条件，Mysql要求每从聚簇索引中取出一条记录都要判断是否符合边界条件，若不符合则释放掉锁，并返回（返回之前不会释放1～5的S锁）。
    3. 最后符合要求的id依次为，1～5。
    4. 但是其在判断是否符合条件时是要加锁的，若不符合条件会在释放锁(若该记录有锁则会阻塞)。

  2. 若id=6此时存在锁,则该查询会阻塞

    1. 若条件为id>=5，则在最后一条Supremum伪记录并不会上锁
    2. 并不时根据索引找到id=5的记录，然后向前寻找

  • SELECT ... FOR UPDATE，X锁

  • 若UPDATE语句并没有更新二级索引列，加锁方式与SELECT ... FOR UPDATE语句一致，X锁。若更新二级索引列，UPDATE team_number SET name = '6佚名' WHERE id >= 6;，则会先对聚簇索引加X锁，在对二级索引加锁。

    1. 先对id=6的记录的聚簇索引加锁，再对idx_name索引加锁
    2. 对id=7的记录重复上述操作
    3. 对所有满足条件的记录重复上述操作。
    4. 若条件为id<=7，则只会对聚簇索引加锁，随后释放锁，并不会对二级索引加锁

  • DELETE ...，会对聚簇所有以及所有二级索引加锁，其过程与UPDATE ...一致

二级索引进行等值查询

在READ UNCOMMITTED和READ COMMITTED隔离级别下，使用普通的二级索引和唯一二级索引进行加锁的过程是一样的。

  • SELECT ... FOR UPDATE;。例如SELECT * team_number FORCE INDEX(idx_name) WHERE name = '6佚名' FOR UPDATE; ，其是先对idx_name所属的二级索引加X锁，然后进行回表操作(不考虑数据量太小，Mysql直接全表查询的情况)，对聚簇索引加X锁。
    • 当二级索引非唯一索引时，MySQL对等值匹配的条件有特殊处理，他们规定在InnoDB存储引擎层查找到当前记录的下一条记录时，在对其加锁前就直接判断该记录是否满足等值匹配的条件，如果不满足直接返回
    • 事务1先对二级索引加锁，同时事务2则对同一记录的聚簇索引加锁，这种情况会导致死锁。
  • SELECT ... LOCK IN SHARE MODE。与SELECT ... FOR UPDATE;类似。先对二级索引加S锁，在对聚簇索引加S锁。
  • UPDATE ...，与SELECT ... FOR UPDATE;类似,若更新了其他二级索引的字段，则也会对该索引加锁。
  • DELETE ...，与UPDATE ...类似

二级索引进行范围查询

  • SELECT ... LOCK IN SHARE MODE;为记录加S锁，先给二级索引加S锁，再给聚簇索引加S锁。
  • SELECT ... FOR UPDATE;来为记录加X锁，先给二级索引加X锁，再给聚簇索引加S锁。
  • UDPATE ...，与SELECT ... FOR UPDATE;类似,若更新了其他二级索引则会对其加锁
  • DELETE ...与``UDPATE …`类似
  • UPDATE team_number SET sex = 1 WHERE number <=127;，其从前向后查询然后比较，对每一条记录都是先加锁后判断(索引条件下推或者返回)，在5.7.21中，并没有释放掉最后一个记录的锁(number=128)

在使用主键进行范围查询时，已经说明其会对最后一条记录先加锁后释放锁(若update包含二级索引，则也不会对二级索引加锁)，但是针对二级索引进行范围查询时，number=128记录是先对二级索引加锁，在对聚簇索引加锁，然后释放掉聚簇索引锁，在释放二级索引锁

全表扫描

  • SELECT ... LOCK IN SHARE MODE;为聚簇索引记录加S锁

  • SELECT ... FOR UPDATE;为聚簇索引记录加X锁

    在遍历聚簇索引过程中，都会先为每条记录加锁，若不满足条件则释放

  • UDPATE ...，与SELECT ... FOR UPDATE;类似,若更新了其他二级索引则会对其加锁

    若不满足条件，也不会为二级索引加锁

  • DELETE ...与``UDPATE …`类似

3. 幻读

MySQLREPEATABLE READ要解决的是幻读问题，而幻读是新增记录，所以不涉及幻读(插入SQL)的加锁方式均与READ UNCOMMITTED和READ COMMITTED一致

GAP锁的S锁与S锁目标一致，都是为了解决幻读，禁止插入记录。

主键进行等值查询

  • SELECT * FROM team_number WHERE id = 9 FOR UPDATE;，
    • 主键具有唯一性，所以多次执行改SQL时，也不会出现别的记录，也就没有幻读问题，此时与READ UNCOMMITTED和READ COMMITTED一致，在id=6的记录上加X锁。
    • 若id=9的记录不存在,那么为了禁止幻读产生，必须要在(7,20)之间加gap锁

其他情况不会产生幻读，所以与READ UNCOMMITTED和READ COMMITTED一致。

主键进行范围查询

  • SELECT * FROM team_number WHERE id >= 8 FOR UPDATE，为了解决幻读问题，需要禁止插入id >=8的记录，而主键是唯一的,所以只需保证禁止插入id>8的记录就可以。
    • 为id=8的记录加上X锁
    • 为id>8的所有聚簇索引加上X型的Next-Key锁(Record锁+GAP锁)
    • 针对Supremum记录也加GAP锁，但是当作GAP锁处理的。
  • SELECT * FROM team_number WHERE id <= 7 FOR UPDATE，这个有点特殊(与READ UNCOMMITTED和READ COMMITTED类似)
    • REPEATABLE READ隔离级别下，在判断id=18的记录不满足边界条件id<=7后，并不会去释放加在该记录上的锁
      • 不这里为什么要这样
  • SELECT ... LOCK IN SHARE MODE与SELECT ... FOR UPDATE类似
  • UPDATE ...，与SELECT ... FOR UDPATE类似，如果UPDATE语句更新二级索引列，则需要将对应二级索引列加锁
    • update team_number set name = '匿名' where id <= 7，虽然会对id=18的聚簇索引加锁,但是并没有对其二级索引加锁
  • DELETE与UPDATE类似

唯一二级索引进行等值查询

  • SELECT * FROM team_number WHERE number=126 FOR UPDATE
    • number列是唯一二级索引，所以没有幻读问题，只需要给idx_number加一个X型锁以及对聚簇索引加X锁。
    • 若number=130，数据不存在，则会对number=149加GAP锁，但是并不会给聚簇索引加锁
  • SELECT ... LOCK IN SHARE MODE，与SELECT ... FOR UPDATE类似
  • UPDATE ...与SELETE ... FOR UPDATE类似,若被更新的列包含其他二级索引列，则对应的二级索引记录也会被加X型正经记录锁。

唯一二级索引进行范围查询的

  • SELECT ... FOR UPDATE，与使用主键进行范围查询类似，区别是先对二级索引加GAP锁，再对对应的聚簇索引记录加RECORD锁
    • SELECT * FROM team_number where number >= 147 FOR UPDATE;，对number >= 147加Next-Key锁

  • SELECT * FROM team_number WHERE number <= 129 LOCK IN SHARE MODE;其会给number=129以及之前的每条记录加Next-Key锁，其会对id>129的第一条记录加Next-Key锁并不会释放
  • SELECT ... LOCK IN SHARE MODE与SELECT ... FOR UPDATE类似
  • UPDATE ...与SELECT ... FOR UPDATE类似，若更新了其他二级索引，则也会对该索引列加锁
  • DELETE ... 与UPDATE ... 类似

普通二级索引进行等值查询

  • SELECT ... LOCK IN SHARE MODE，例如SELECT * FROM team_number where name='匿名' FOR UPDATE;
    • 普通二级索引，所以允许重复，其会对所有name='匿名'的记录加Next-Key锁，然后对最后一个name='匿名'的下一条记录加GAP锁。
    • 若普通二级索引不存在，则会在其下一条记录上加Gap锁
  • SELECT ... LOCK IN SHARE MODE与SELECT ... FOR UPDATE类似
  • UPDATE ...与SELECT ... FOR UPDATE类似，若包含其他二级索引，则会对对应索引记录加X锁
  • DELETE ...与UPDATE ...类似

普通二级索引进行范围查询

与唯一二级索引情况一致

全表扫描

  • SELECT * FROM team_number where sex = 2 FOR UDPATE,全表扫描，每读取一条记录都会先加X锁，再判断sex = 2条件是否成立

在REPEATABLE READ隔离级别下，InnoDB存储引擎并不会真正的释放掉锁，所以聚簇索引的全部记录都会被加锁，并且在事务提交前不释放。此时其他事务的任何请求都会被阻塞，会严重影响并发事务处理能力。

  • SELECT ... LOCK IN SHARE MODE与SELECT ... FOR UPDATE类似
  • UPDATE ...与SELECT ... FOR UPDATE类似，若包含其他二级索引的也会对二级索引加锁
  • DELETE与UPDATE ...类似，且会对所有二级索引加锁

INSERT 语句

INSERT语句一般情况下不加锁，不过当前事务在插入一条记录前需要先定位到该记录在B+树中的位置，如果该位置的下一条记录已经被加了gap锁(next-key锁也包含gap锁)，那么当前事务会在该记录上加上一种类型为插入意向锁的锁，并且事务进入等待状态。

主键或者唯一索引重复

  • 当插入重复主键时报错，Duplicate entry '20' **for** key 'PRIMARY',其实在报错之前会对聚簇索引中加S锁(相当于一次读取)

    • 在READ UNCOMMITTED/READ COMMITTED隔离级别下，加的是S型正经记录锁。
    • 在REPEATABLE READ/SERIALIZABLE隔离级别下，加的是S型next-key锁。

  • 当插入唯一的二级索引重复时，会在报错前对已经在B+树中的唯一二级索引记录加Next-Key锁。

  • INSERT ... ON DUPLICATE KEY ...，会对B+树中已存在的相同键值的记录加X锁，而不是S锁。
